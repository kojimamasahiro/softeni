# 詳細検討: データによる選手プレースタイルのプロフィール化

日付: 2026-07-11
状態: Draft（詳細検討。親: [2026-07-11-idea-player-style-profiling.md](./2026-07-11-idea-player-style-profiling.md)）
位置づけ: 「個人が追いかけたくなる指標」の具体化。ユーザーが「面白そう」「分析手法を研究したい」と反応した案。

## 1. 誰に刺さるか（ターゲット分析）

診断・タイプ分け系のコンテンツは「**自分（推し）は何型か**」という自分ごと化が刺さりの核。層によって別の顔を持たせる。

| 層 | 刺さる理由 | 露出面 |
|---|---|---|
| **記録ユーザー本人**（選手個人） | 自分のプレーが数値の「形」になる。勝敗以外の自己認識ツール。「自分はネットで決めるタイプ」が可視化される | score機能の分析画面（private。ADR-003 のUGC private 既定で公開論点なし） |
| **指導者・チーム** | 選手のタイプ把握・ペア編成（前衛/後衛適性、タイプ相性）の判断材料。B2B2C 収益案とも整合 | score機能（グループ内。ADR-004 L1 の範囲） |
| **showcase 閲覧者**（一般読者） | トップ選手の「型」を見る楽しみ（ATP公式が6分類を「共通言語」にしているのと同じ観戦文法の提供） | `/growth/[slug]` ショーケース（運営キュレーション1〜2選手のみ。ADR-004 allowlist 内） |

主対象は**記録ユーザー本人と指導者**（= score機能の獲得・定着ドライバー）。一般読者向けはショーケース経由の「デモ」として機能させ、「自分もこのプロフィールが欲しい」→ score 登録の導線にする。**この案自体が score CTA の中身になれる**（ADR-004 でプレースホルダのまま保留されている CTA に載せる具体的価値）。

## 2. 多くのユーザーに刺さるための設計原則

1. **強さとスタイルを混同しない**（ハーバード研究の最重要教訓）。スタイル軸には勝率系でなく**構成比**（どう決めたかの内訳）を使い、強さ系指標（得点率）は別枠で見せる。
2. **公開面ではポジティブ表現のみ**。「エラーが多い」でなく「攻撃的」。弱点の言語化は private（本人・グループ内）に限定。未成年比率の高い競技での実名能力評価には ADR-004 の慎重方針をスタイル表示にも適用する。
3. **ラベルを付けるのはデータが許してから**。Phase 1 は「形を見せる」だけ（レーダーチャート）。型ラベルの誤認定（n が小さいのに「守備型」と断定）は本人の反発を生む最悪パターンなので、段階ゲートを厳守する。

## 3. 実装構成

### 3.1 Phase 1: スタイル・レーダーチャート（今すぐ可能）

クラスタリングなし・ラベルなし。既存 point データから軸を算出して形だけ見せる。

**軸の設計（スタイル4軸＋強さ2軸を分離表示）**

| 区分 | 軸 | 算出 |
|---|---|---|
| スタイル | ネット決定志向 | 自得点中の `volley_winner`＋`smash_winner` 構成比 |
| スタイル | ストローク決定志向 | 自得点中の `passing_winner` 構成比 |
| スタイル | 短期決着志向 | 自ポイントの `rally_count` 1-2球の構成比（`rally_count>0` のみ分母） |
| スタイル | 粘り志向 | 自ポイントの `rally_count` 5球以上の構成比 |
| 強さ | サーブ局面力 | サーブ時得点率 |
| 強さ | 勝負強さ | 重要局面（デュース・ゲームポイント）得点率（`lib/matchAnalysis.ts` の局面判定を再利用） |

- スタイル軸は構成比なので勝率と数学的に独立（原則1を構造で担保）。
- 正規化: **記録済み全選手の分布に対するパーセンタイル**を 0-100 に写像（データセット内相対）。母数が小さいうちは分布が不安定なので、チャートに「掲載記録◯試合分」の注記（scope-limited と同じ表記規約）を必ず添える。
- 表示ゲート: **最低3試合かつ100ポイント以上**の選手のみプロフィール生成（現データで該当は上松俊貴・黒坂卓矢・上岡俊介＋α。ショーケースのデモにはこれで足りる）。
- 実装: `lib/styleProfile.ts`（純関数: points[] → StyleProfile）。`lib/growthAnalysis.ts`・`lib/matchAnalysis.ts` と同格の共有資産として、score（private）と showcase（public）の両面で再利用（ADR-003 の共有エンジン方針）。

```ts
type StyleProfile = {
  subjectKey: string;                 // growthAnalysis の targetKey 規約と同一
  sampleSize: { matches: number; points: number; ralliesRecorded: number };
  styleAxes: { key: string; percentile: number; raw: number }[];  // 構成比ベース
  strengthAxes: { key: string; percentile: number; raw: number }[]; // 得点率ベース
  confidence: 'scope-limited';        // 常に掲載範囲注記
};
```

### 3.2 Phase 2: ルールベースの型ラベル（データ拡大後）

- 発動条件（案): 表示ゲートを満たす選手が **30人以上**になったら着手。
- 閾値ルール例: ネット決定志向が75パーセンタイル超→「ネットプレー型」等。ATP 6分類のソフトテニス翻訳ラベル案: **ネットプレー型 / ストローク型 / 速攻型 / ラリー型 / オールラウンド型**（前衛/後衛というポジション自体が既にスタイル語彙として競技に根付いているので、ポジション内の細分として意味が出る語彙を選ぶ。要ユーザー感覚チェック）。
- ラベルは常に「◯◯型の傾向」と傾向表現に留め、public 面では強さ軸と切り離して表示。

### 3.3 Phase 3: クラスタリング（研究フェーズ）

- 発動条件（案): **100人以上×各5試合以上**（それでもハーバード研究の桁には届かないため、あくまで研究として）。
- k-means＋シルエット分析で k を選定。スタイル軸のみを特徴量にし、強さ指標は入れない（フェデラー問題の回避）。Phase 2 のルールベース分類と突き合わせ、一致度が高ければラベル体系を維持、乖離すればラベル再設計。
- 型相性分析（カウンター型はビッグサーバーに有利、の類）はランキング/レーティングでスキルを統制する必要があるため、Elo 系レーティング導入後の題材（open-questions のランキング副指標と連動）。

### 3.4 フェーズまとめと接続

| Phase | 発動条件 | 出すもの |
|---|---|---|
| **P1** | 今すぐ | `lib/styleProfile.ts`＋レーダーチャート。score 分析画面＋ショーケース1〜2選手 |
| **P1.5** | P1後 | ショーケースに「あなたのプロフィールも作れる」CTA（ADR-004 のプレースホルダに実体を配線） |
| **P2** | ゲート通過30人 | ルールベース型ラベル |
| **P3** | 100人×5試合 | クラスタリング研究・型相性分析 |

### 3.5 KPI

- P1: score 分析画面でのプロフィール閲覧率、ショーケースページの滞在時間変化。
- P1.5: ショーケース→score 登録のコンバージョン（この案の本丸。「個人が追いかけたくなる指標」が獲得ドライバーとして機能するかの検証）。

## 4. 課題・未解決（更新）

- `rally_count` の記録品質: basic 記録で 0（未記録）のポイントがあり、ラリー系2軸の分母が痩せる。`ralliesRecorded` を sampleSize に持たせ、少なすぎる場合はラリー軸を非表示にするフォールバックが要る。
- パーセンタイル正規化は母集団が変わると本人の値が動く（記録が増えると「ネット志向60→45」等）。本人向け表示では raw 値併記で緩和するが、見せ方は P1 で要検証。
- `opponent_level` による補正は P1 では見送り（注記のみ）。相手レベルでスタイル数値がどう歪むかは P2 前に検証。
- シングルス/ダブルスの混在: 前衛/後衛でスタイル軸の意味が変わるため、ダブルスはポジション別に分けるか、当面シングルス＋個人集計で始めるか。P1 実装時の最初の決定事項。
