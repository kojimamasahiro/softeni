# score機能の一般公開・課金展開 検討メモ（2026-07-11）

Claude との対話ログ。score機能を「一般の人が練習試合などでも使える」形へ広げ、
将来の課金・ピボット先の候補を出すための発散フェーズの調査記録。
まだ何も決定していない（Draft Ideas のみ）。

## Summary

- score機能（試合記録・動画レビュー・成長分析）を一般公開・課金対象に広げたい、という出発点から、
  市場規模・競合・記録文化・実データでの相関分析まで調査した。
- 結論: 需要はゼロではないが競合はすでに存在し単価も低いニッチ市場。softeni-pickの優位性は
  「記録係不要（動画ベース事後記録）」「大会・選手データベースとの接続」「重要局面別の分析エンジン」の3つ。
  どれを主軸にするかは未決定。
- 実データ（16試合・717ポイント）でパイロット分析を行い、ソフトテニス特有の構造的示唆
  （サービスエースがほぼ出ない、ブレークポイントの非対称性、短ラリーのサーブ有利など）が
  実際に取れることを確認した。これは分析エンジンの説得力を示す材料になる。

## 調査の流れ

1. 課金機能としての発展方向のアイデア出し
2. 一般ユーザー（練習試合）向け展開の需要規模の検討
3. 野球とソフトテニスの「スコアを記録する文化」の有無の比較
4. 硬式テニスとの比較
5. 野球でのスコア活用法（スカウティング・個人成長）の確認
6. 硬式テニスにおけるスタッツと勝敗の相関の確認
7. 自リポジトリの実データ（`public/data/beta-matches/matches/*.json`）でのパイロット相関分析

## Draft Ideas（課金・展開の候補、優先順位未決定）

### 差別化の核（3候補、択一ではなく組み合わせもあり得る）

1. **動画ベースの事後記録**: 試合中に記録係を置かなくても、動画から後でポイント候補を抽出できる
   （`match_video_sessions` / `match_point_candidates` は実装済み）。ソフトテニスは野球と違い
   マネージャー的な記録専業ポジションが部活動に制度化されておらず、この「記録係不要」設計が
   競合の手動記録アプリに対する優位点になり得る。
2. **大会・選手データベースとの接続**: 既存の大会結果・ランキング・選手ページという資産と、
   練習試合の記録・分析を紐づけられる。競合アプリ（データテニス等）は単体ツールでこれを持たない。
3. **重要局面別の分析エンジン**: `lib/growthAnalysis.ts` / `lib/matchAnalysis.ts` は単純平均でなく
   デュース・ゲームポイント等の局面別指標を持つ。硬式テニス研究でも「平均値より局面のタイミングの方が
   勝敗説明力が高い」とされており、方向性は理にかなっている（下記パイロット分析で裏付け）。

### 差別化の核（続き・2026-07-11後半のセッションで追加）

4. **普及エンジンとしての接続**: softeni-pickにはすでに`historical-winners`/`milestone`/`career-record`という、
   公式大会データから「初優勝」「連覇」「通算N勝」「王者撃破」を自動抽出し記事化するパイプラインが
   ADR-005（news-context-block-architecture）として設計・実装されている。これは野球の「記録の永続化→
   成績算出→新聞というメディアでの流通」と同型の構造。ただし現状の入力は`data/tournaments/details/**`
   （公式大会結果）のみで、score機能の試合データ（`public/data/beta-matches/**`）とは別系統。
   両者を接続すれば、日常のプレーからも記事化できるネタが生まれ続ける。
   - 公式大会の結果データは勝敗・順位しか持たないが、score機能のpoint-levelデータは
     「どう勝ったか」を持つ。これはscore機能だけが提供できる物語のネタになる。

### 発展候補アイデア: 希少イベント検知 → SNS/ニュース投稿（メジャー大会・手動記録ベース）

**まだ発散フェーズ。詳細を詰めていない。今後発展させられそうな案の1つとして記録するのみ。**

- 想定運用: メジャーな大会が開催されたら、YouTube配信からscore機能の動画レビュー機能
  （既存の`match_video_sessions`/`match_point_candidates`）で手動記録する。記録したポイント列から
  「希少な状態」を検知し、動画リンク付きでSNSに投稿、あるいはニュース記事化する。
- 運営が選ぶ公式大会の手動記録に限定すれば、すでに記事化・実名公開している範囲と同じなので、
  UGC練習試合の`visibility=private`既定（ADR-003）や未成年配慮（ADR-004）の論点をほぼ回避できる。
  これが「メジャー大会限定」というスコープの絞り方の一番の利点。
- パイロットチェック（16試合・109ゲーム・717ポイント、n小さく参考値）:
  「珍しい」と直感していたカテゴリのうち、実際に希少だったのはサービスエース（717中2件、0.3%）と
  ロングラリー（21球以上が717中4件、0.6%）のみ。2点差以上の逆転勝ち（109ゲーム中12件、11%）や
  完封ゲーム（同15件、13.8%）、デュース到達（同29件、26.6%）は想定より頻繁で、閾値の調整
  （逆転は3点差以上に絞る、完封はファイナルゲーム限定にする等）か、大会単位の相対評価
  （「この大会で一番長いラリー」等、絶対閾値でなく大会内順位で選ぶ）が必要そう。
- 動画リンクは動画クリップの生成・再配布ではなく、`video_start_ms`から
  `youtube.com/watch?v=XXX&t=123s`形式のタイムスタンプ付きURLを生成する方式が、
  著作権リスクが低く実装コストも小さいという方向性。
- 検知の粒度は将来的にポイント単位・ゲーム単位・試合単位・大会単位（複数試合の横断集計）の
  4段階に拡張できそうで、選手個人の記録（milestone/career-record連携＝上記4番の差別化の核）とも
  接続しうる。
- 未解決の実務課題: 記録労力に対して希少イベントの歩留まりが低い可能性（厳しい閾値だと
  1試合あたり0件のことも多い）。`shot_type`/`shot_course`はスキーマにあるが実データでは未収録
  （現状は`recording_level: "basic"`のみ運用）なので、より細かい希少性判定にはこの記録UI自体の拡充が要る。

### 収益化オプション（4案、優先順位未決定）

1. 個人向けフリーミアムSaaS（記録数・動画レビュー時間に無料枠、ADR-003のscore.softeni-pick.com
   Phase 2と直結）
2. チーム/クラブ向けB2B2C（顧問・コーチが複数選手をまとめて記録・比較。個人課金より単価を上げやすい）
3. スカウティングレポート販売（対戦相手の傾向分析を大会前に一回課金で提供。既存の大会・選手データ資産を転用）
4. 既存AdSense/アフィリエイトのscore側への拡張（補完策、影響小）

一番筋が良さそうなのは1（ADR-003がすでに向いている方向）。2・3は差別化が強いが検証コストが高い。

### 見送り候補（Rejected / Deprioritized）

- **実名の全体公開・ランキング型ショーケースを課金導線にする**: ADR-004で「全体公開（L2/L3）は
  当面保留」と既に決定済み。今回の課金検討とは切り離す。
- **記録文化を啓蒙する形の一般啓発コンテンツ課金**: 野球のような記録文化がそもそも薄い市場で
  啓蒙コストが高すぎる可能性。動画ベース事後記録で「記録する手間」自体を無くす方が優先度が高いと判断し、
  今回は深掘りしなかった。

## 調査結果の詳細

### 市場規模・競合

- 日本ソフトテニス連盟登録会員数は約40万人、うち中学生が約26万人（65%）。中学→高校で大きく減少。
  「競技人口700万人」はプレー経験者を含む推定値で課金対象の目安にはならない。
- 既存競合: データテニス（テニス/ソフトテニス両対応、プレミアム300円/月、無制限記録・CSV出力等）、
  ソフトテニス パーフェクトスコア（ソフトテニス専用、111項目自動分析）、スコアNK、ソフトテニス手帳。
  いずれも数年運用されており需要の存在は裏付けられるが、価格帯（300円/月）が示す通り支払意欲は低い。
- softeni-pick本体はすでに大会結果・ランキング目的の読者を持っており、新規集客より既存読者の
  コンバージョンの方が現実的な導線。

### 記録文化の比較（野球 / ソフトテニス / 硬式テニス）

- 野球のスコアブック文化はアメリカ発祥を輸入し、直木松太郎・飛田穂洲が意図的に「記録を残す」目的を掲げ、
  新聞・学生野球界という組織的経路で全国標準になった。加えて①一球ごとに区切りがある試合構造、
  ②マネージャーという記録専業ポジションの制度化、③プロ中継・新聞のスタッツ文化、が定着を支えている。
- ソフトテニスはこの3つが薄い（連続的なラリー、記録専業ポジションなし、国際プロツアー放送文化がほぼ無い）。
  「なぜ記録しないか」は単なる気分的慣例というより、記録文化を定着させる歴史的な仕掛けが
  そもそも存在しなかった、という構造的な説明の方が近いという整理（＝Assumption、聞き取り未検証）。
- 硬式テニスはATP/WTA中継のスタッツ表示文化があり、個人主導で記録する層（VersuStats開発者が
  年間40大会に出るアマチュアだった例）が一定数いる。ただし競合（VersuStats、データテニス）が
  すでに刺さっている市場でもある。
- 野球のスコア活用法: 初球の振り方・打球方向からの打者傾向分析、カウント別傾向からの配球戦術、
  相手チームスカウティング（映像併用のデジタルスコアブックアプリ、高校野球現場で年間3万円程度の
  本格アプリ利用例あり）、個人・チームの成長記録としての日記的活用。

### 硬式テニスのスタッツと勝敗の相関（先行研究）

- ファーストサーブ成功率と勝率の相関、ブレークポイント奪取率（BPC）の重要性、アンフォーストエラー率が
  安定性の指標になること、は各種分析記事で確認。
- ただし「ファーストサーブ70%神話」は福岡大学の学生による機械学習分析で通説ほどの差は出なかった
  という反証あり。また平均値より「いつそのプレーが起きたか」（重要局面での成否）の方が
  勝敗説明力が高いという指摘もある（クレイ4-5での凡ミス連発の例）。

### 自リポジトリ実データでのパイロット分析（2026-07-11実施）

対象: `public/data/beta-matches/matches/*.json` 全16試合、109ゲーム、717ポイント
（うち`status: completed`は5試合）。singles/doublesはほぼ均等（346pt / 371pt）。

手法: `lib/matchRules.ts`のルール（通常ゲーム4点先取・ファイナル7点・2点差条件）で各ポイント時点の
累積スコアを再構成し、1ポイント目・2-2後・デュース・ゲームポイント（サーブ側/レシーブ側）を判定。
Python（pandas/numpy、scipy不可のため二項比率の簡易z検定のみ）で集計。

結果:

- サーブ側の総合得点率は51.7%（717pt中）。硬式テニス男子プロ（60%超が珍しくない）より弱い優位性。
- result_type内訳: service_ace は0.3%（2/717）のみ。決着の主因は`out`(24.4%)・`net`(17.2%)などのエラーと
  `passing_winner`/`volley_winner`（合計26.8%）で、サーブで押し切るというよりラリーで決まる競技特性が
  データに表れている（Assumption: ソフトテニスのアンダーハンドサーブ特性による）。
- 1stサーブフォルトの有無による得点率差はほぼ無し（52.5% vs 51.6%、z=0.22、n=159で非有意）。
  硬式テニスの通説と逆の結果だが、サンプルが小さく仮説レベル。
- ラリー長別サーブ側得点率: 1-2球=74.6%、3-4球=46.8%、5-8球=50.2%、9+球=50.9%。
  最初の1-2球で決着すればサーブ側有利、ラリーが続くとほぼ五分五分になる構造。
- サーブ側自身のゲームポイントでは51.7%→56.7%に上昇。レシーブ側がゲームポイントを握る局面
  （＝ブレークチャンス）ではサーブ側得点率が39.0%まで低下＝レシーブ側は61%の確率でブレークを
  ものにしている計算（n=82）。硬式テニス研究の「ブレークポイントが勝敗を左右する」という知見と方向が一致。
- 完了済み5試合すべてで、勝者の自分のサーブ時得点率が敗者のそれを上回っていた（差+14.4pt〜+35.2pt、
  例外なし）。n=5と小さいが最も説得力のある結果。
- 交絡チェック: singles/doublesでサーブ側得点率はほぼ同一（51.7% vs 51.8%）。draft/completedステータス間で
  やや差（53.2% vs 47.0%）があるが試合ごとのばらつきの範囲内と見るべき。

**注意**: N=717ポイント・16試合はごく小さいサンプルで、一部は`status: draft`。デュース(n=41)や
ダブルフォルト(n=8)のような細分化した指標は信頼区間が広い。厳密な有意性検定はしていない。
「本格研究」ではなく「score機能に蓄積されたデータで実際に何かが言えることを示すパイロット」という位置づけ。
