# ハイスクールジャパンカップ2026 1日目SNS投稿 改善設計ドラフト

日付: 2026-06-16
状態: 設計議論メモ（実装前）。出力は議論・設計のみ。実装着手は未定。
関連: `tools/sns-images/`（day1_results.py / power_map.py / h2h.py / snslib.py）

## 目的

2日制で行われる各種目の「1日目終了時点」にXへ投稿し、2日目の**集客**につなげる。
方針: **手元データ（内部）から客観的事実として出せるものは全て自動化する**。
外部データ（配信URL・コート割・リアルタイム進行など）に依存するものは別途検討し、安易に自動取得しない。

## 大会スケジュール（2026・ユーザー提供。要・一次ソース照合）

- 6/26（金）9:00–17:00 シングルストーナメント（1回戦〜**準々決勝終了まで**）／13:00–公開練習
- 6/27（土）9:00– シングルス準決勝〜決勝（終了後シングルス表彰式）→ 表彰式後 **ダブルス予選リーグ（12面）開始**
- 6/28（日）9:00– ダブルス決勝トーナメント（決勝まで）／15:00頃 ダブルス表彰式・閉会式
- 注: 荒天時はダブルス予選リーグをトーナメントに変更する場合あり。
- 開催地・開催日・主催ソースURLは内部 `data/tournaments/information/highschool-japan-cup.json` に存在（=内部データ）。

## このスケジュールから確定する「1日目」の意味（重要）

当初想定（ダブルス1日目=リーグか？）を実スケジュールで補正した結果:

- **シングルス1日目（6/26金 夜）= ベスト4確定**。準々決勝まで終了 → 残存4。
  - データは knockout のみ（`singles-none-*.json` で確認済み）。**現行 day1_results.py で対応可**。
  - 勝ち残りドローは「準決勝ブラケット（4→2→1）」と小さく、結果一覧は1〜準々決勝。
- **ダブルス1日目（6/27土 夜）= 予選リーグのみ**。
  - データは roundrobin + knockout の2段（`doubles-none-*.json` で確認済み）。
  - 1日目終了時点で確定するのは「**リーグ星取・グループ順位・本選進出ペア**」。
  - **現行 day1_results.py はリーグ未対応** → ここが最大のギャップ。新規描画が必要。
- 投稿タイミングは実質2回（金夜＝シングルス／土夜＝ダブルス）。
  - 土曜は二重負荷: ダブルス1日目投稿に加え、**シングルスは土曜で完結**するため結果報告投稿も発生し得る。

## データ棚卸し（何が客観的に出せるか）

内部データのみで判定。シード情報は **存在しない**（entry/participant に seed なし）。

| コンテンツ                                  | データ源                         | 自動化                             |
| ------------------------------------------- | -------------------------------- | ---------------------------------- |
| 残存勢力サマリ（県別/校別 残存数・ベストN） | details JSON                     | ◎ 完全自動                         |
| 2日目の確定カード一覧（準決勝など）         | details JSON                     | ◎ 完全自動（"確定対戦"は客観事実） |
| 勝ち残りドロー・回戦別結果一覧              | details JSON                     | ◎ 既存 day1_results.py             |

## コンテンツ設計（客観事実ベース）

主観的な「注目」ラベルは付けず、客観事実を並べて事実に語らせる方針。

1. **残存勢力サマリ（自動）**: 県別残存数、校別残存数、ベストN到達。シングルス1日目なら「ベスト4の4選手＋所属県」。
2. **2日目の確定カード（自動）**: ブラケットから「両者確定済み・未消化」の試合を抽出＝客観的な"2日目の対戦"。シングルスは準決勝2カード。
3. **番狂わせ＝過去実績proxy（要定義）**: シードが無いので、**「前年（または過去N年）ベスト4以上だったペア/選手が初日で敗退」**を客観イベントとして抽出。逆に「前年王者が生存」も生存事実として併記可。`results[].tournament.rank` を年跨ぎ参照（participant id で名寄せ）。
   - 留意: ペア構成は年で変わるため、ダブルスは「選手単位」で前年実績を引く設計が必要（要相談）。
4. **CTA（内部データ範囲で自動）**: 「2日目: ◯月◯日 9:00〜 @会場（北海道 等）」「主催: gosen-sp.jp」まで自動。配信URL・コート割は外部のため載せない（または手動メタで差し込む）。
5. **リーグ表（ダブルス1日目・新規）**: グループ別星取表＋順位＋「本選進出 ○ペア」を描画。

## 運用設計

- **投稿はカテゴリ別**（男子D/女子D/男子S/女子S）。Xは1投稿4枚まで → 各投稿の4枚構成を定型化。
  - 例（シングルス1日目）: ①CTA ②残存サマリ（ベスト4） ③準決勝ブラケット ④確定カード/番狂わせ
  - 例（ダブルス1日目）: ①CTA ②本選進出サマリ ③リーグ表(1〜2枚) ④番狂わせ/注目進出
- **キャプション自動生成**: details/information から客観テンプレ＋ハッシュタグを生成。
  - 例: 「【男子S 1日目終了】ベスト4が決定。◯◯県勢が△名生存。2日目は6/28(日)9:00〜 @北海道。#ハイスクールジャパンカップ #ソフトテニス」
- **一括実行ラッパ**: 4カテゴリをまとめて生成。1日目の「段（リーグ/トーナメント）」をカテゴリごとに自動判定して出力種別を切替。

## 外部データの扱い（要検討フラグ）

- 配信URL・開始時刻詳細・コート割は手元に無い。
- 選択肢:
  - (a) 自動化対象外。CTAは内部データ（日付・会場・主催URL）までに留める。
  - (b) 大会ごとの軽量手動メタ（例 `tools/sns-images/meta/hjc2026.yaml` に配信URL等）を任意で差し込む。
  - 外部サイト(gosen-sp.jp)からの自動取得は規約・安定性の観点で別途検討（現時点では非推奨）。

## 決定事項（2026-06-16 確定）

- スケジュール一次ソース照合: **不要**（ユーザー提供スケジュールを採用）。
- ダブルスの番狂わせproxy: **選手単位の前年実績で引く**（ペア再編に対応するため、ペアではなく選手 id 基準）。
- 過去実績proxyの基準: **前年のみ**／閾値 **ベスト8以上**。
  - すなわち「**前年ベスト8以上に入った選手が、今年の1日目で敗退**」を番狂わせイベントとして抽出。
  - 生存側併記も同基準（前年ベスト8以上が生存）。
- CTA: **内部データ(a)に限定**。日付・会場・主催URLまで。配信URL・コート割・開始時刻詳細は載せない（手動メタは導入しない）。
- 荒天時ダブルス=トーナメント変更: **自動判定は実装しない**（必要時に手動対応）。

## 4枚構成テンプレ（確定 2026-06-16）

投稿はカテゴリ別。X=1投稿4枚まで。種目・段に応じて以下を定型化する。

### A. シングルス1日目（金夜・男子S/女子S 各1投稿）= ベスト4確定

1. 表紙＋CTA：「男子S 1日目終了｜ベスト4決定」＋ 2日目 6/28(日)9:00〜 @北海道（CTAは内部データ範囲）
2. ベスト4一覧（4選手・所属県/校）
3. 準決勝 確定カード（2カード）← 集客の主役
4. 1日目ハイライト：**準々決勝結果（後半のみ・集客重視）** ＋ 前年ベスト8以上の敗退/生存（番狂わせproxy）を強調表示
   - 1回戦〜2回戦の全結果一覧は出さない。

### B. ダブルス1日目（土夜・男子D/女子D 各1投稿）= 予選リーグ終了

1. 表紙＋CTA：「男子D 1日目終了｜決勝トーナメント進出ペア決定」＋ 2日目 6/28(日)9:00〜
2. 本選進出サマリ（進出ペア数・県別/校別進出数）
3. **本選進出ペア一覧（進出ペア重視・集客向き。1枚、収まらなければ2枚）**
4. 番狂わせproxy（前年ベスト8以上の予選敗退/通過）
   - リーグ星取表は省略。詳細星取が要る場合のみ別途検討（集客投稿には載せない）。
   - → リーグ「表」描画は必須要件から外れ、優先度1は「**進出ペア確定の抽出・描画**」に変更。

### C. シングルス完結報告（土・男子S/女子S 各1投稿）= 集客ではなく記録/反応狙い（設計に含める）

1. 優勝（選手・所属を大きく）
2. 準優勝・ベスト4
3. 決勝・準決勝のスコア
4. （任意）締め／県別実績などの総括（内部データ範囲）
