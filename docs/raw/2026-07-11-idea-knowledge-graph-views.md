# アイデア: Knowledge Graphによるデータ設計・UX統合(view-based開発)

## 状況

Idea Backlog。発散フェーズ（2026-07-11 検討開始）。まだ設計・実装には入っていない。

## 目的

現状、選手・チーム・大会・試合の「関係」を扱うロジックが機能ごとに個別実装されている。
新機能を1つ足すたびに専用の関係解決ロジックが増えていく現状の開発スタイルから、
「エンティティと関係をグラフとして持ち、新機能は基本的にグラフ上の新しいビュー（見方）を
追加するだけで済む」開発スタイルに寄せられないか、というデータ設計とUXの両方にまたがる検討。

## ユーザーが興味を持った点

> Knowledge Graphを使ったデータ設計やUXの向上

> 機能を増やすたびに専用ロジックを追加するのではなく、グラフ上の新しい見方（ビュー）を追加する
> という開発スタイルに近づけられる可能性があるのではないかと考えている

きっかけの温度感を聞いたところ「どちらも」（＝既存の重複実装を見た保守性の課題感、かつ
新規UX機能を作る際にグラフがあれば楽そうという両方の動機）との回答。

## 想定運用 / なぜこのスコープか

まだ発散フェーズにつき未定。ただし対話の中で、いきなり大規模リファクタに入るのではなく、
「グラフとして見たら本当に楽になるか」を1つの具体的なビューで検証してから判断する、という
順番自体には合意している（既存データだけで作れるものから着手する方針）。

## わかっていること

コードベース調査（2026-07-11実施）で、エンティティ間の関係を逆引き・再構築するロジックが
既に複数箇所で独立実装されていることを確認した。

- `lib/matchReverseIndex.ts`（大会/選手 → score試合の逆引き）と
  `lib/playerStats/reverseIndex.ts`（選手 → 出場カテゴリの逆引き）は、どちらも
  「逆引き索引を1回だけ構築してプロセス内キャッシュする」という同型のパターンだが、
  別実装・別idスキームで存在する。
- 選手idの解決だけでも `resolveNumericId` / `resolveAliasedPlayerId` / `homonyms.json` /
  `playerKey`（氏名@所属）と、複数の名寄せ経路が並行して存在する
  （詳細: [team-player-identity.md](../wiki/team-player-identity.md)）。
- 選手/チーム/大会/試合の関係を大会データから毎回別ロジックで再構築しているファイルが多数ある
  （行数はいずれも2026-07-11時点）: `careerRecord.ts`(280行) `milestones.ts`(479行)
  `majorTitles.ts`(200行) `tournamentRecords.ts`(406行) `growthAnalysis.ts`(1118行)
  `matchAnalysis.ts`(1664行) `newsArticle.ts`(1262行)。`lib/` 全体で11,236行。
- `lib/playerStats/facts.ts` は対戦相手（`opponents`）を既に選手ごとに集計しており、
  H2H的なエッジ情報は実質的に既存データの中にある。
- 既存の暗黙のノード/エッジ構造を整理すると概ね以下になる:
  - ノード: Player・Team・Tournament(+年+カテゴリ)・Match
  - エッジ: `participated_in`(Player-Match)、`belongs_to`(Player-Team、年で時限)、
    `played_against`(Player-Player、Matchから導出=H2H)、`teammate_of`(Player-Player、ダブルス)、
    `same_person_as`/`homonym_of`(Player-Player、名寄せ)

## 課題・未解決

- 具体的なグラフスキーマ（ノード/エッジの正式定義）はまだ設計していない。
- グラフをどう持つか（既存の静的JSON生成パイプラインの上に論理層として被せるだけか、
  実体としてグラフ構造のデータストア/インデックスファイルを新設するか）は未検討。
- 既存libの大規模リファクタに広げるか、まず1つの新規ビューだけを試作して効果を検証するかは、
  試作結果を見てから判断する方針（現時点では未決定）。
- どのビューを最初のプロトタイプにするか自体は次の3案が対話中に出たが、まだ選定していない:
  1. 対戦相手ネットワーク（選手ページに「よく当たる相手・勝率」をグラフ表示。
     `facts.ts` の `opponents` を流用できそうで着手コストが低い）
  2. チーム/学校の系譜（`team-player-identity.md` の名寄せ結果を使い、
     ある選手が中→高→大とどのチームを経てきたかを辿る）
  3. プレースタイル類似ネットワーク（バックログの
     [プレースタイルのプロフィール化案](2026-07-11-idea-player-style-profiling.md)と接続し、
     似たスタイルの選手同士をエッジで結ぶ。新規計算が必要でコストは高め）

## 目指したい方向性

まだ発散フェーズ。次の一歩としては、上記3候補のいずれかを小さく試作し、
「グラフとして扱うことで専用ロジックが実際に減らせるか」を検証してから、
データ設計側（既存libのグラフ層への統合）に広げるかどうかを判断する、という流れになりそう。
