# 詳細検討: ランキング較正ハーネス

日付: 2026-07-11
状態: Draft（詳細検討＋パイロット実施済み。親: [2026-07-11-idea-ranking-calibration-harness.md](./2026-07-11-idea-ranking-calibration-harness.md)）

## 1. 問題設定

現行のランキング設計（`ranking-config.json`: tier 100/60/20・順位係数・上位3大会合算）は
「運用開始後に実データで微調整する」とされているが、**何をもって「良くなった」と判定するかの
指標が無い**。これが「精度がイマイチ」の正体。係数をいじっても良否が分からなければ較正は進まない。

解決策: ランキングの唯一の客観的な検証手段は「**上位認定した側が実際に勝つか**」。
過去の全対戦結果をバックテストの正解データとして使い、予測的中率を精度の定義にする。

## 2. パイロット結果（2026-07-11実施・生数値）

素朴なEloを既存全データに走らせ、ハーネスの成立性を確認した。

**抽出**: 411ファイル中、使用可能27,667試合（除外: 団体戦等のentries≠2が2,786、
勝者未記録372、選手名未解決1,222、非dict形式63）。ユニーク選手（名前ベース）18,415人。

**手法**: 名前ベース同定（同姓同名融合を許容＝プロジェクト既定方針と同じ）、一律K=32、初期1500、
ダブルスはペア平均レートで期待勝率を計算し両選手に同デルタ、`retired`は全除外、
時系列は 年→大会→ファイル→ブラケット深度（`prevMatchIds`のトポロジカル順）で近似。

**結果**:

| 指標 | 値 |
|---|---|
| 予測的中率（レート差あり全対戦） | 64.1%（16,027/25,020） |
| 予測的中率（両者5試合以上） | **68.6%**（5,087/7,418） |
| 5試合以上の選手 | 5,909人（10試合以上2,423人） |

**較正曲線**（両者5試合以上）:

| Elo差 | 的中率 | n |
|---|---|---|
| 0-49 | 54.8% | 2,964 |
| 50-99 | 69.6% | 1,838 |
| 100-149 | 78.9% | 1,085 |
| 150-199 | 80.4% | 622 |
| 200-249 | 86.7% | 368 |
| 250-299 | 92.0% | 225 |
| 300+ | 95.6% | 316 |

**解釈**: 較正曲線が単調に伸びており、レート差が勝敗確率として意味を持つことを実データで確認。
この曲線がそのまま「番狂わせの定量定義」になる（例: 期待勝率15%以下の勝利=ジャイアントキリング）。
年間試合数は2024年7,872→2025年10,007と増加中で、精度は放っておいても上がる基調。

**パイロットの既知の限界**（ハーネス本実装で解消すべき点）:

- 大会の年内順序が未考慮（`information/**` の開催日で並べ直す）
- 一律K（tier別K・provisional期間は未適用。config骨組みには定義済み）
- 名前融合（homonyms.json の16名は成績が混ざる。既定方針通り許容だが評価時に除外も可）
- 団体戦（entries≠2）を落としている（個人対戦に展開できるかは別途）

## 3. ハーネスの構成

```
data/tournaments/details/**（正解データ: 27,667試合・毎年約1万増）
  ↓ scripts/ranking/backtest.mjs（新規）
  │   ①時系列再生: information/** の開催日で大会を並べ、ブラケット順に試合を再生
  │   ②評価対象を差し替え可能に:
  │      - 現行ポイント式ランキング（その時点のランキング順位で勝者予測）
  │      - Elo副指標（レート差で勝者予測）
  ↓ metrics JSON（的中率・較正曲線・カバレッジ・年別推移）
  ↓ 係数チューニング: ranking-config の候補値グリッドを回し metrics を比較
```

**設計上の要点**:

- **評価の切り出し**: 「設定→ランキング計算→予測→採点」を純関数化し、config候補を差し替えて
  何度でも回せるようにする（`lib/playerStats` の集計と密結合にしない）。
- **過学習ガード**: 年でホールドアウトする（例: 2024年以前で係数を選び、2025年以降の的中率で
  最終評価）。全期間に係数を合わせ込むと「過去に最適・今後に弱い」設定を選んでしまう。
- **評価対象のゲート**: 両者の既知試合数（例: 5試合以上）で層別して報告する。新規選手だらけの
  対戦の不的中はランキングの罪ではないため、混ぜると改善が見えなくなる。
- **メトリクスは2本立て**: 的中率（分かりやすい）＋較正誤差（Elo理論勝率と実測勝率のズレ。
  giant-killing の確率定義に使うのはこちら）。

## 5. ポイント式とEloの役割分担（提案）

- **ポイント式ランキング**（公開の主指標): 説明可能・大会実績ベース。読者向け。
- **Elo**（内部/副指標): 予測・希少性判定エンジン。giant-killing、シード妥当性、
  スタイル分析のスキル統制（プレースタイル案のフェデラー問題対策）に使う。
- 両者の乖離自体もコンテンツになりうる（「実績よりレートが高い=急上昇中の選手」等、
  「個人が追いかけたくなる指標」への還元）。

## 6. 課題・未解決

- 大会開催日の取得率（`information/**` に日付が無い大会の順序フォールバック）。
- 団体戦の扱い: 個人対戦として展開すれば試合数がさらに増えるが、オーダー情報の構造次第。
- 名前融合の評価への影響: homonyms 該当選手を評価から除外した場合の的中率差分を P1 で計測。
- 年またぎのレート減衰（学生競技は卒業・引退で母集団が入れ替わる。シーズン間で
  provisional に戻す/減衰させる設計は P3 で検証）。
- ポイント式ランキング自体の的中率測定（Eloとの比較ベースライン）は P1 の最初の成果物。

## 7. P1 実装状況（2026-07-11 追記）

`scripts/ranking/backtest.mjs` を実装した。実行: `node scripts/ranking/backtest.mjs [options]`。

- §3の設計どおり: 開催日順再生（`information/**` の startDate、日付なし大会は年内末尾）、
  tier別K（`ranking-config.json` の kByTier。tier判定は isMajorTitle→major / index.json掲載→national /
  非掲載→local）、ブラケット順再生（prevMatchIds トポロジカル順）。
- 評価対象は `--evaluator elo|points|both`。points は生成済み `data/rankings/**` を
  **前年度スナップショット**として使用（Assumption: 対戦時点で参照可能なのは前年度ランキング）。
- オプション: `--k-mode tier|uniform` / `--k` / `--min-prior` / `--holdout-year` /
  `--provisional-boost` / `--exclude-homonyms` / `--out <metrics JSON>`。
- Assumption: international 系大会は専用tierが無いため national の K で扱う。

**初回計測結果**（holdout-year 2025）:

| 設定 | 的中率(両者5試合以上) | holdout(2025+) |
|---|---|---|
| Elo tier別K | 69.0% (5,016/7,268) | 70.4% |
| Elo 一律K=32 | 69.0% | 70.4% |
| Elo tier別K + provisional×2 | 68.7% | 70.2% |
| Elo tier別K + homonym除外 | 69.1% | 70.5% |
| ポイント式（前年度掲載の対戦のみ） | 67.2% (2,882/4,288) | 67.5% |

わかったこと:

1. **Eloがポイント式を上回り、カバレッジも桁違い**（ポイント式は両サイド前年度掲載の対戦が
   全体の16.9%しかない。前年度ランキング非掲載＝新人・掲載外選手の対戦を予測できない）。
   Elo副指標の有効化価値は的中率よりむしろカバレッジにある。
2. **年を追うごとに精度上昇**（2022: 61.1% → 2026: 70.5%）。「データが増えるほど精度が上がる」
   基調を実測で確認。
3. **較正曲線で実測が理論を一貫して上回る**（例: Elo差50-99で実測70.3% vs 理論60.3%）。
   レートが実力差に対して圧縮されている＝**Kが小さすぎる or スケール(400)が広すぎる**示唆。
   次の較正はKの引き上げ方向から探るのが良さそう（P2の最初の実験）。
4. tier別Kと一律Kの差は現状ほぼ無し。provisional boost(×2)はやや悪化。homonym除外の影響は
   +0.1pt程度で、名前融合の許容方針は精度面でも当面問題ない。

## 8. P2 較正結果（2026-07-11 追記）

`--sweep-elo`（K×scaleスイープ）と `--grid-points`（tier比・順位係数・topNグリッドサーチ）を
ハーネスに追加し実行した（holdout-year 2025）。ポイント式グリッドは details の placement から
シーズンポイントを再計算するミラー実装（`rankingCompute.ts` と同式。名寄せ簡略のため絶対値に微差）。
現行configの再現値（fit 66.6% / holdout 67.6%）が data/rankings ベースの実測（67.2% / 67.5%）と
ほぼ一致し、ミラーの妥当性を確認済み。

### 方法論上の発見: 的中率は K/scale にほぼ不感

的中率は「レート差の符号」しか使わないため、K や scale を変えても順序がほぼ保存され動かない。
確率較正の良否は **Brierスコア**（確率誤差の二乗平均、小さいほど良い）で測る必要があり、
ハーネスに追加した。giant-killing の「期待勝率◯%以下の勝利」定義に効くのはこちら。

### Elo: K/scale 比 0.16 が較正最良（現行の2倍）

| K/scale | Brier(fit) | Brier(holdout) | 的中率(holdout) |
|---|---|---|---|
| **0.16**（例: K=64, scale=400） | **0.2076** | **0.1929** | 70.4% |
| 0.08（現行相当: K=32, scale=400） | 0.2125 | 0.1985 | 70.4% |

§7の「実測が理論を上回る＝Kが小さすぎる」仮説をBrierで定量確認。**推奨: kByTier を2倍**
（major 80 / national 64 / local 48）。的中率は変わらず、確率の信頼度だけが改善する。

### ポイント式: topN=2・flat係数・national比0.45 で +1pt 前後

| 設定 | fit | holdout |
|---|---|---|
| 現行（coef current / national 0.6 / topN 3） | 66.6% | 67.6% |
| 最良（coef flat{1, .8, .6, .4, .2} / national 0.45 / topN 2） | 68.5% | 68.4% |

傾向: topN=2 > 3、flat係数（entry寄与を上げる）が優位、national比は0.3〜0.45が0.6より良い。
local比は details が index.json 収録大会のみのため実質未検証（差が出ない）。

**ただし反映は保留判断**: ポイント式は読者向けの「年度実績ランキング」であり、係数は
予測精度だけでなく「何を実績として讃えるか」という商品設計を決める。flat係数は参加寄与を
重くする方向で、実績表彰としての性格を変える。予測精度の +1pt は Elo 副指標で既に上回れる
（69.0%・カバレッジ全選手）ため、**予測はEloに任せ、ポイント式係数は実績表彰の観点で決める**
のが役割分担（§5）に沿う。

### config 反映

- `rating.kByTier` を {major 80, national 64, local 48} に更新（2026-07-11。`rating.enabled` は
  false のままなので生成物への影響なし。有効化時の初期値として較正済みの値を置く趣旨）。
- ポイント式係数は上記の理由で未変更（ユーザー判断待ち）。

## 9. 設計論点: ポイント式 vs レーティング vs ハイブリッド（2026-07-11 追記）

P2の「ポイント式係数を変えるか」から派生した論点。ユーザー発言:
「ポイント式係数をどうするかは悩ましいところである。別の観点としてレーティング形式にした場合は
どうか？それともレーティングを考慮したポイントにするかどうかということか？」

### 3つの設計の整理

| 設計 | 答える問い | 評価 |
|---|---|---|
| 1. ポイント式（現行） | 今季誰が一番実績を残したか | 説明可能・シーズン中に下がらない・未成年公開に安全。予測力は劣る（計測済み） |
| 2. レーティング形式に置換 | 今誰が一番強いか | 予測・カバレッジ最良だが、負けると下がる数字の実名公開（ADR-004と地続き）、説明困難、参加ディスインセンティブ。**主指標としては不採用**（ATPも公式は実績ポイント制、Eloは分析側） |
| 3a. 獲得ポイントを相手レートで補正 | 混合 | 説明可能性が崩れ実績表彰の性格も変わる。悪いところ取りになりがちで慎重 |
| 3b. tier重みをレートで客観化 | 実績（重みだけ客観化） | 式は現行のまま、tier係数の恣意性を解決する案 → 下記実験 |

結論の方向: **主指標=ポイント式（実績）、Elo=副指標（予測・希少性・急上昇検出）** の役割分担（§5）を維持。

### 3b実験の結果（`--rated-tier`、holdout-year 2025）

tier重みを「大会参加者の事前レート平均」（時系列Elo再生の大会開始時点スナップショット、先読みなし。
3試合以上の選手のみ・10人未満の大会はフォールバック。61/66大会で算出可）から導出して評価した。

| tier重みの与え方 | fit | holdout |
|---|---|---|
| 静的・P2最良（100/45/10 + flat + topN2） | **68.5%** | 68.4% |
| 動的 rated:exp200 + flat + topN2 | 67.3% | 67.7% |
| 動的 rated:exp400 + flat + topN2 | 67.2% | 67.2% |
| 静的・現行（100/60/20 + current + topN3） | 66.6% | 67.6% |

**予測面では動的tierは調整済み静的tierに勝てない**（現行configよりは僅かに良い程度）。
重みが毎年動く説明可能性コストを払う価値はなく、**tier重みの動的置換は不採用**。

### ただし副産物: 大会強度の導出は「tier誤設定の監査ツール」として有効

導出された大会強度の上位に、現行tierと大きく食い違う大会が出た:

| 平均レート | 大会 | 現行tier |
|---|---|---|
| 2054 | lucent-tokyo-indoor 2025 | **local (20/100)** |
| 1961 | zennihon-indoor 2025 | major（妥当） |
| 1903 | asian-games-qualifier 2025 | **local** |
| 1870 | zennihon-senbatsu 2026 | national |
| 1767 | yonex-hokkaido-international 2025 | **local** |

resolveTier の「国際・国際予選→isNational=false→local」ルールが、少数精鋭の招待制・代表選考系
大会を最弱tierに置いており、明確なミスプライシング。ルーセント東京インドアが local 扱いは直感にも反する。

**採用したい形**: レーティングの知見は「式への組み込み」でなく「**定期監査→静的tierの手動再分類**」
で還流させる。静的tierの説明可能性を保ったまま、レート由来の強度で誤設定を検出する運用。
ハーネスの `--rated-tier` 出力（上位大会テーブル）がそのまま監査レポートになる。

注意（監査指標の限界）: 参加者平均レートは「少数精鋭」に有利で、規模・層の厚さ（優勝難度）を
測らない。再分類判断では field の平均だけでなく規模も見ること（将来は top-16平均等の併記を検討）。

### 次のアクション候補

- elite招待・国際系大会の tier 再分類（monetization でなく resolveTier / index.json 側の設計判断。
  再分類後に P2 グリッドを再実行して静的tierの最適値を取り直す）。→ §9.1 で実験済み
- ポイント式係数（flat + topN2 で +1pt）は実績表彰としての性格変化を許容するかのユーザー判断待ち（§8から継続）。

## 9.1 国際大会の除外とtier再分類の実験（2026-07-11 追記）

ユーザー方針: 「国際大会は外国の選手はデータがないので怪しくなるから思い切って除外してもいい」。

### 対象の切り分け（データ実体確認済み）

| 大会 | 実体 | 扱い |
|---|---|---|
| international-korea-cup | 参加452名中ローマ字名420（外国選手＋日本選手もローマ字で名寄せ不能） | **除外**（gen='international'） |
| international-hiroshima-peacecup | gen='international' | **除外**（同上） |
| 国際予選3つ（world/asian-championship/asian-games qualifier） | **全員国内選手**（都道府県あり・漢字名） | 除外せず**再分類** |
| lucent-tokyo-indoor / yonex-hokkaido-international | 全員国内選手。**index.json 非掲載**のため local 扱いだった | 除外せず**再分類** |

ハーネスに `--exclude-international`（gen='international' を再生・評価から除外）と
`--retier` / `--retier-national`（精鋭大会を major / national へ）を追加。

### 結果（グリッド最良行 flat・national0.45・topN2 で比較、holdout-year 2025）

| 条件 | fit | holdout |
|---|---|---|
| 除外のみ（再分類なし） | 68.5% | 68.4% |
| 除外＋再分類→national | 68.5% | 68.6% |
| **除外＋再分類→major** | 68.4% | **69.3%** |
| （参考）現行config＋除外＋再分類→major | 66.5% | 68.2% |

- データは**再分類→major を支持**（holdout +0.9pt。係数を変えない現行configでも +0.6pt）。
  国際予選・ルーセントの優勝は「majorタイトル同等の重み」とする方が過去の勝敗と整合する。
- 再分類→national は中間（+0.2pt）。「代表予選はタイトルではない」という表彰観をとるならこちら。
- 監査（--rated-tier）の再実行で、強度上位10大会のtierがすべて major/national に揃うことを確認。

### 本エンジンへの反映（2026-07-11 適用済み）

ユーザー決定: 「除外＋再分類→major を反映、係数は現行維持（予測はElo副指標に任せる役割分担を維持）」。

1. **国際大会の除外**: `ranking-config.json` に `excludeTournaments`
   （international-korea-cup / international-hiroshima-peacecup）を追加し、
   `computeSeasonPoints`（rankingCompute.ts）でシーズンポイント算入から除外。
   facts は変更なし（勝敗・H2H等の統計には残る。除外はランキングのみ）。
2. **tier再分類**: `ranking-config.json` に `tierOverrides` を追加（国際予選3つ＋lucent→major、
   yonex-hokkaido→national）。`entryScore` が resolveTier より優先して適用。
   config ローダ（config.ts）に型・既定値・`_note`キー除去を実装。
3. **係数**: 現行維持（winner 1.0 / runnerup 0.7 / best4 0.5 / best8 0.3 / entry 0.1、topN=3）。
4. `data/rankings/**` を `--full` 再生成。テスト28件パス。2026年度からローマ字名の孤立レコードが
   消滅し、再生成後のポイント式の holdout 的中率は 67.6%→**68.1%**（係数据え置きで +0.5pt）。
   2025男子ダブルスの上位も変動（lucent・国際予選の major 重み反映）。
5. ハーネス（backtest.mjs）は config の tierOverrides / excludeTournaments を常時ミラーするよう
   更新（エンジンとの整合を維持。--retier / --exclude-international は追加実験用として温存）。

注意: engineVersion（facts層）は据え置き。facts 自体は不変で、ランキングは rankings 生成時に
facts から再計算されるため。lucent-tokyo-indoor / yonex-hokkaido-international の index.json
掲載可否は別途のコンテンツ判断として未解決のまま（tierOverrides は非掲載でも機能する）。
