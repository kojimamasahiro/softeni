# アイデア: 希少ポイント/ゲームイベント検知 → SNS/ニュース投稿

## 状況

Idea Backlog。発散フェーズ、未着手（2026-07-11時点）。詳細設計はしていない。

## 目的

score機能の試合記録データを、単なる個人向け分析ツールとしてだけでなく、
**ソフトテニス自体の露出を増やす普及エンジン**としても使えないか、という発想。
記録したポイント列の中から「珍しい状態」を見つけ、動画リンク付きでSNS投稿・ニュース記事化することで、
大会に来ない人にも競技の面白さを届けたい。

## ユーザーが興味を持った点

3案の中で、ユーザー（masahiro）が最初に「1つやりたい」と名指しした案。原文:
「希少な状態のデータを検知してそれをニュースか単にsnsに投稿するかみたいなのは1つやりたい。
これがベースとして、発展できるかを検討する１案にしたい」。他の2案（ランキング連動、選手プロフィール化）は
この案を掘り下げる過程で派生的に出てきたもので、出発点としての優先度はこれが一番高い。

## 想定運用

1. メジャーな大会が開催されたら、YouTube配信からscore機能の動画レビュー機能
   （既存の`match_video_sessions` / `match_point_candidates`、`src/pages/beta/matches/[matchId]/video-review.tsx`）
   で手動記録する。
2. 記録したポイント列から「希少な状態」を検知する。
3. 検知したイベントを、動画リンク付きでSNSに投稿する、あるいはニュース記事化する。

## なぜ「メジャー大会・運営による手動記録」に絞るか

- 一般ユーザーのUGC練習試合はADR-003で`visibility=private`が既定方針。ADR-004も未成年配慮から
  実名の全体公開に慎重（ソフトテニス競技人口の65%が中学生）。
- 運営が選ぶ公式大会の手動記録に限定すれば、**すでに大会結果として実名公開している範囲と同じ**なので、
  UGCのプライバシー論点をほぼ回避できる。これが今のところ一番現実的な入り口。

## わかっていること（パイロットチェック、2026-07-11実施）

`public/data/beta-matches/matches/*.json`（16試合・109ゲーム・717ポイント、n小さく参考値）で
候補カテゴリの出現頻度を確認した。

- **実際に希少だったもの**: サービスエース（717中2件、0.3%）、ロングラリー（21球以上が4件、0.6%）。
- **直感より珍しくなかったもの**: 2点差以上の逆転勝ち（109ゲーム中12件、11%）、完封ゲーム
  （同15件、13.8%）、デュース到達（同29件、26.6%）。これらは絶対閾値のままだと投稿しすぎて
  特別感が薄れる、あるいは対戦相手（多くは未成年）への配慮の点でも扱いが難しい。
- 示唆: 絶対閾値でなく、**大会単位の相対評価**（「この大会で一番長いラリー」等）にする方が
  歩留まりが安定しそう、という仮説段階。

詳しい集計過程・生数値は [2026-07-11-score-general-availability-research.md](./2026-07-11-score-general-availability-research.md)
の「発展候補アイデア: 希少イベント検知」節を参照。

## 技術的な方向性（仮）

- 動画リンクは動画クリップの生成・再配布ではなく、point の `video_start_ms` から
  `youtube.com/watch?v=XXX&t=123s` 形式のタイムスタンプ付きURLを生成する方式が、
  著作権リスクが低く実装コストも小さい。
- 検知の粒度は将来的にポイント単位・ゲーム単位・試合単位・大会単位（複数試合の横断集計）の
  4段階に拡張できそう。

## 課題・未解決

- **歩留まり**: 手動記録という労力に対して、厳しい閾値だと希少イベントが少ない（1試合あたり0件の
  こともありうる）。カテゴリ拡充や大会単位の相対評価で改善できるかは未検証。
- **記録の粒度不足**: `shot_type` / `shot_course` はスキーマにあるが実データでは未収録
  （現状 `recording_level: "basic"` のみ運用）。より細かい希少性判定にはこの記録UI自体の拡充が要る。
- SNS投稿先プラットフォーム、投稿前レビューの要否は未検討。

## 目指したい方向性

- まずポジティブに語れるカテゴリ（エース、ロングラリー、僅差からの逆転勝ち）から、
  安全に自動化しやすいものとして始めたい。
- 完封のように片方を下げる形になりうるカテゴリは、自動公開でなく人がレビューする運用が無難。
- 選手個人の記録（milestone/career-record連携）と将来的に接続できると、単発の投稿でなく
  「その選手の物語」として蓄積できるはず（別案「[勝敗・大会単位の希少性検知](./2026-07-11-idea-giant-killing-ranking.md)」とも接点あり）。

## 実装（2026-07-11 追記）

P0 を実装した。現状仕様は [docs/wiki/rare-events.md](../wiki/rare-events.md)（実装が source of truth）。

ユーザー決定（2026-07-11）:

- 「検知イベントを SNS 投稿 or ニュース記事化」のうち、サイト表示は**試合詳細ページに
  スコア記録とともに載せる**形を採用（原文:「いったん、スコア詳細ページにスコアの記録とともに
  載せるのがいいかと思っている」）。当初の「大会ごとのニュースに追記」は、ADR-010 で result 記事が
  廃止済みだったため見送り。大会結果ページ等への差し込みは P2 候補として残す。
- SNS への投稿は**手動**（レポートスクリプトが出すテンプレ文をコピペ）。
- データが少ないうちの誤検知は**許容**（scopeNote 表示で担保）。
- 「何をもって希少とするか」は**ハーネス**（`RARE_EVENT_CONFIG` ＋収穫レポート
  `npm run rare-events:report`）で改善を回せる形にした。

## 発見ハーネスへの拡張（2026-07-11 追記）

ユーザーの追加要望（原文）:
「希少データのハーネスは決められた条件での大小で決めるだけでなく、新たな発見ができるようにしたい。
たとえば複数のデータを組み合わせたら希少データであったなど。例えば、後衛がスマッシュでポイントを
取ったなど。ただ、それは条件を複雑にしただけの可能性もあるため、その複雑さをどこまで許容するのかを
調整する基盤にできるといい」

実装（詳細は [docs/wiki/rare-events.md](../wiki/rare-events.md) の discovery 節）:

- 特徴量の組み合わせ探索＋**独立仮定の期待件数**による分類（発見候補 / サブパターンに還元可能 /
  条件を複雑にしただけ）。`npm run rare-events:discover`。
- 複雑さの許容度は `DISCOVERY_CONFIG` のノブ（maxConditions / minExpectedCount /
  maxLift＋条件数ペナルティ）で調整する基盤にした。
- 発見候補は人がレビューし、`RARE_EVENT_CONFIG.patterns` に昇格すると既存パイプライン
  （サイト表示・投稿テンプレ）に自動で乗る。
- 制約: 例に挙がった「後衛」ロールは現行データに存在しない（shot_type/shot_course も未収録）。
  記録されるようになれば特徴量に自動合流する設計。初回実行では
  「デュース局面をスマッシュで決めた（期待4.7件に対し実測1件）」等が発見候補に挙がった。

## 判定スコープの変更（2026-07-12 追記）

ユーザーの要望（原文）:
「この大会の名場面のコンテンツがあるが、記録できない試合の方が多いので、大会ごとではなく
全体の中での判定にしてほしい」

対応: 比較スコープを「大会開催単位」から**記録済み全試合の横断比較（all-time）**に変更した。
記録できる試合が大会のごく一部である以上、「この大会最長」は分母が薄く実態と合わないため。
`RARE_EVENT_CONFIG.scope` で切替可能とし、'tournament' は当初案として温存（記録カバレッジが
上がったら再検討）。ラベルは「記録済み全試合で最長の〜」、ページ見出しは「この試合の名場面」へ。
同一データでイベントは 24件→8件に絞られ、1件あたりの特別感は上がる。プランの§2.1
「大会単位なら必ず1件成立（歩留まりの構造的解決）」は、全体判定でも「プール内最長は必ず存在する」
形で保たれる（ただし新規大会を記録してもイベントが出ない場合はある＝サイト記録更新時のみ）。
