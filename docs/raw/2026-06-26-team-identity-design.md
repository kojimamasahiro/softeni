# チーム名名寄せ・チーム/選手識別 設計メモ

状態: Draft（討議メモ。確定は wiki / ADR 化の段階で行う）
日付: 2026-06-26

## 背景・課題

大会結果データ（`data/tournaments/details`）のチーム名に表記揺れがあり、名寄せ
したい。現状の揺れは3種類:

1. 半角全角・スペースの揺れ（例: `能登1st` / `能登1ｓｔ` / `能登１ｓｔ`）
2. 省略・冗長表記の揺れ（例: `高田商` / `高田商業` / `高田商業高等学校`）
3. 同名別校（例: `大分商` と `大分`）— 似ているが**統合してはいけない**

規模感（2026-06-26 時点の実測）:

- 異なるチーム文字列: 約 5,152
- 参加者行: 約 44,428 / 大会スコープ 約 20
- NFKC＋スペース除去だけで 161 文字列 → 155 グループに収束（= (1) は機械的に自動化可能）
- 同姓同名で複数チーム名を持つ選手: 約 4,864（ただし多くは表記揺れでなく正当な所属違い）

以前、高校カテゴリで「同姓同名を集め、文字列が連続していれば同一校・していなければ
別校」と手動で切り分けた。これを全体へ広げるか別手法かを検討した結果が本メモ。

## 既存の状況（重要）

リポジトリには既に2つの識別方式が混在している。

- 方式A（tournament 側）: `data/tournaments/team-name-aliases.json` が**文字列**を
  canonical にしている（現状 9 校のみ）。さらに participant id =
  `姓_名_team_都道府県` で、**チーム文字列が選手 id に埋め込まれている**。
- 方式B（highschool 側）: `data/highschool/teams.json` と
  `data/teams/team-name-mappings.json` は**不変の人工キー**を源流にしている
  （例: `himejishou`, `nssu`）。表記は属性にすぎない。

選手側も2層になっている:

- 出場単位の id（`姓_名_team_都道府県`）= 「人」ではなく「ある大会への出場」の単位。
- 人単位の curated slug（`data/players/<slug>/`、例 `yano-soto`）= 氏名の NFKC 正規化で
  解決（`resolveSlugByName`）。**team を含まない**。通算成績はこちらを軸に集計。

## 決定事項

### D1. 源流は「文字列」にしない

識別キー（行を同一チームとして束ねる軸）と表示名（ユーザーに見せる文字列）は求める
性質が違う。文字列を源流にすると両者が癒着し、表示を直すたびに id が崩れて
差分・リンク・SEO リダイレクトが連鎖する。よって**識別は不変 id、表記は属性**とする。

### D2. 一般チーム = 単純連番 id

`data/tournaments/details` 全スコープ（`highschool-championship` 等も含む）に現れる
実チームを**連番 id** で識別する。名寄せ（5,152 文字列）の対象はこの系統。

- `id`（連番）が識別の源流。一度採番したら不変。
- 表記揺れは `aliases` に集約。半角全角・スペースは機械正規化で前処理してから入れる。
- `canonicalName` は表示用の代表表記。**最頻出表記を機械的に採用**する（NFKC＋スペース
  正規化後に、その id 配下で出現回数が最多の表記。同数は文字数が短い方など決定的タイブレーク）。
  後で変えても id は不変。
- マスタ配置: `data/teams/teams.json`（選手マスタ `data/players/index.json` に倣う）。

連番（不透明 id）を選ぶ理由: 発番が機械的で衝突ゼロ、ローマ字化の読み曖昧
（中央=chuo/nakaou 等）を持ち込まない。URL の見栄えより運用の堅さを優先。

### D3. `/highschool/` 機能のチーム = 既存ローマ字 slug、別系統で維持

「高校カテゴリ」とは**`/highschool/` ページ配下の機能**を指す（大会ジャンルではない）。
`data/highschool/teams.json`（`himejishou` 等、約 333 校）はそのまま、独立した別系統と
して維持する。大会ジャンルで切らないため、高校名が高校外スコープ（全日本・東日本など、
実測 59 校）に出る件は本系統の論点にならない。

### D4. 紐付け（ただし畳まない）

一般チーム（連番）から `/highschool/` 機能のチーム（ローマ字 slug）へ任意リンクを張る。
同じ学校でも**レコードは別物として両方保持**し、辿れるようにするだけ。クラブ・社会人・
大学・小中など高校ページに無いチームはリンク無し（null）。

## データモデル案

### 一般チームマスタ（新規・連番）

```jsonc
{
  "id": 1,
  "canonicalName": "高田商",
  "prefecture": "奈良県",
  "aliases": ["高田商業", "高田商業高等学校", "大和高田市立高田商業高等学校"],
  "highschoolSlug": "takadashou",   // /highschool 機能への任意リンク。無ければ null
  "note": "OB団体『高田商ＯＢクラブ』は別チーム"
}
```

- `id` 連番・不変。`aliases` に表記揺れを集約。`canonicalName` は表示用。
- 県は名寄せのブロッキングキーにも使う（別県の「商業」を自動で別物にできる）。

### `/highschool/` 機能マスタ（既存・ローマ字）

`data/highschool/teams.json` を現状維持（`id: "himejishou"`, `name`, `prefecture`）。

### 紐付け

一般チームの `highschoolSlug` → 高校マスタ `id`。双方向に辿れる。同一実体でも
両レコードを維持（自動マージしない）。

## 選手 id（年度で team が変わる問題）

### 結論: team 変更は「識別」の問題ではなく「関連」の問題

人 id に team を埋めない。

- **人 id（連番・不変）= 源流**。team を含まない。
- **所属 = 出場レコードの属性**。`(人id, teamId, 年, 大会)` の関係として記録する。
- team が年度で変わる ＝ **同じ人 id に所属レコードが複数ぶら下がるだけ**。identity は
  1 つのまま。現状も通算成績が氏名で名寄せして集計しているため、この形と整合する。

### 注意: team-in-id が兼ねていた「同姓同名の判別」を分離する

現状 `姓_名_team_都道府県` で team/県を入れているのは、表記の安定だけでなく
**同姓同名の別人を区別する**役割も兼ねている（田中翔が 2 人いても team/県で分かれる）。
team を id から外すなら、この切り分けを別途持つ必要がある。これは以前の高校カテゴリの
手作業（同姓同名を集めて切り分け）と同じ問題。curated slug 側は「氏名が一意なら解決、
曖昧なら null」と割り切って回避している。人レジストリ（連番人 id ＋ 氏名 ＋ 別人フラグ）で
一度切り分け、以後は再利用する方針とする。

### D5. 選手 id の移行（Open Question 解決・2026-06-26）

調査でわかった現状:

- 人単位の数値 id は**既に存在**する。`data/players/index.json`（数値 id ↔ (姓,名)、
  約 8,182 人、team 非依存）。URL `/players/[id]` も既にこの数値 id で動く。
- composite id（`姓_名_team_都道府県`）は**詳細 JSON 内の局所キー**にすぎない。URL にも
  `redirects-map.json` にも無く、`participants[]` と `entries[].playerIds` をファイル内で
  繋ぐだけ。よって移行は URL 層を壊さない。
- composite 形式には既にばらつきがある（県欠落 `田端_一葵_和歌山`、pref 欄に federation が
  入った `房野_紗千_四天王寺高_高体連` 等）。脆い文字列キーの症状。

メリット:

- **名寄せ（team 正規化）が選手 id に波及しなくなる**。今 `normalize-team-names.mjs` は
  team を直すたびに id を再計算して `playerIds` を張り替えている。team を id から外せば
  この連鎖（と巨大差分リスク）が消える。元課題に直結する最大の利点。
- 人の同定が build 時の氏名マッチ（ヒューリスティック）から**確定参照**になる。
- team 変更が純粋に属性化し、人 id は年度・所属に不変。
- composite 形式のばらつきを一掃できる。

デメリット／コスト（地雷）:

- **同姓同名の判別を先に解く必要**。現 `index.json` は (姓,名) で衝突ゼロ＝**別人を 1 id に
  畳む前提**。このまま数値化すると別人の同名が統合され、サイレントなデータ破壊になる。
  以前の高校手作業と同じ切り分けを人レジストリ整備として全体で行う必要がある。
- 取り込みフローに「氏名→人 id 解決（新規採番・同名判定込み）」の前段が増える。
- 海外・非日本選手（`LEI SHANG-JUNG`/TPE 等）も登録対象になる。
- 移行作業量: 279 ファイル・約 44k の id と `playerIds` 参照の書き換え。

実装戦略（2 択）:

- **A. 解決層方式**: 詳細の文字列はそのまま、build 時に人 id へ解決。差分小・可逆・段階導入。
  脆い文字列は残るが、人レジストリ整備（同名分割）を低リスクで先行できる。解決層は既に
  半分存在（build 時の氏名マッチ）。
- **B. 物理置換方式**: 詳細の id を数値へ書換。終状態はクリーン・データが正準。ただし
  一度きりの巨大差分・ほぼ不可逆で、同名切り分けを完全に詰めてからでないと危険。

**結論**: まず A で人レジストリ（同姓同名の分割込み）を固め、安定を確認してから B を検討。
team 名寄せの完了を待たず「選手 id を team から切り離す」だけでも独立した利得がある。
別人判定基準は D6 へ。

### D6. 同姓同名の別人判定基準（2026-06-26）

方針: 分割ルールは「誤マージは防ぐが誤分割を生む」方向。**仮分割 → ペアグラフで再マージ →
残渣のみ手動**、の順で精度を上げる。実測（2026-06-26、複数出場の (姓,名) は 8,758 名）。

仮分割ルール（高精度・保守的）:

- **県が違えば別（事前確率）**: 複数出場名の 61%（5,356 名）が複数県に跨る。共通姓名は
  地域で別人が多く方向は妥当。ただし「転居した同一人物」「社会人＝勤務地/所属団体、代表＝
  別表記といったカテゴリ依存の県表記」で誤分割が出るため、flat ルールではなく**強い事前確率**
  として扱い、後段のマージ信号で覆せるようにする。県は canonical 化した値で比較する。
- **同年・排他的生活段階が違えば別**: 「同年・all 以外でカテゴリ違いは別」の精緻版。
  **`generationId` をそのまま使わない**。`junior` が「小中の大会＋全日本ジュニア（U20 で
  大学生も出場可）」の混在バケツで、生 `generationId` だと 381 名が別人候補になるが、その
  大半が誤分割（例: 村田優斗 junior+university 4 年連続 ＝ 大学生かつジュニア代表の同一人物）。
  大会 id から生活段階を引き直す。
  - **分割に使う段階は 小学 / 中学 / 高校 の 3 つだけ**（決定）。大学・社会人・シニア・
    ジュニア代表は基準に含めない（大人は社会人↔シニアが重複し排他でないため、大学・ジュニアは
    他段階と重複するため）。同年に小/中/高のうち**異なる段階**へ出れば別人候補とする。
  - **隣接段階（小↔中、中↔高）も分割対象に含める**（決定）。3 月→4 月の卒業境界で同一人物が
    両段階に出る誤分割はわずか（実測 小↔中 5 名・中↔高 2 名）で、ペアグラフ再マージと残渣手動で
    回収できる。誤統合（サイレントな破壊）を避ける方を優先。小↔高 は段階が飛び物理的に別人で安全。
  - 段階→大会 id（分割に使う）:
    - 小学: `primaryschool-championship` / `zennihon-primaryschool` /
      `primaryschool-akita-indoor` / `primaryschool-miyagi-indoor`
    - 中学: `secondaryschool-championship` / `zennihon-secondaryschool-versus` /
      `zennihon-secondaryschool-club-pre`
    - 高校: `highschool-championship` / `highschool-japan-cup` / `highschool-senbatsu` /
      `highschool-tokai-senbatsu`
  - 免除（分割に使わない）: 大学（`zennihon-university*`）/ 社会人（`zennihon-workers` /
    `zennihon-business-group` / `zennihon-club`）/ シニア（`zennihon-senior`）/
    `zennihon-junior`（U14/U17/U20 の年齢別で中〜大が混在）/ all / international /
    international-qualifier / ローカル・未登録大会（既定で免除）。

再マージ信号（誤分割の回収）:

- **ダブルスのペアグラフ**（最強）: 実データに 18,685 ペア・17,982 ノード。ソフトテニスは
  ダブルス中心で、県・カテゴリを跨いでも**同じ相棒と組んでいれば同一人物**。連結成分で
  クラスタリングし、仮分割した同名クラスタが共通の相棒で繋がれば再マージする。
  - **閾値（決定）**: 共通の相棒が **1 回でもあれば再マージ**（最大限まとめる方向）。
  - 留意: 相棒も氏名で同定するため、別人同士が同名の相棒を1回だけ共有して誤マージする
    残余リスクはある。実害が出たら閾値（共通相棒の共出場回数 ≥2 等）を引き上げる。
- **チーム遍歴の整合**: 小学クラブ→中学→高校→大学/社会人 と年が増え、地域が連続するか。
- **かな・生年**: curated 選手のみ保有（詳細 JSON には無い）。あれば高精度の補助判定。

運用順序: 姓名でブロック → 仮分割（排他的生活段階＋県の事前確率）→ ペアグラフで再マージ →
残った曖昧のみ手動（旧高校手法を残渣だけに適用）。

## 名寄せの進め方（パイプライン案）

ひとつの手法で全部やろうとせず、3 層に分ける。

1. **機械正規化（前処理）**: NFKC ＋ スペース除去を id 生成の前段に組み込み、(1) の揺れを
   恒久的に自動収束させる。手作業ゼロ。
2. **候補生成（ブロッキング）**: 全ペアではなく**都道府県でブロック**し、その中で文字列類似
   （共通プレフィックス／トークン包含）で候補ペアだけ抽出。人間の確認対象を数百規模に圧縮。
   別県の同名は自動で別物になり誤統合も激減。
3. **人間判断 → alias 集約 → 連番採番**: 判断結果を一般チームマスタの `aliases` に蓄積し、
   `id` を採番。`note` に同名別校の区別を残す。一度の判断が再利用され二度手間にならない。

選手の同姓同名シグナルは捨てず、「カテゴリ＋都道府県で限定した上での補助確認」として使えば、
以前の手法の良さも残せる。

  その差を別チーム判定に使わない（両方が実在の都道府県で、かつ異なる時のみ別チーム）。
  判断が要る個別ケースは `note` に根拠を残す。
