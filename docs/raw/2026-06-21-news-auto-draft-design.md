# 速報・プレビュー 自動ドラフト機能 設計ドラフト（A/B）

日付: 2026-06-21
状態: Draft（2026-06-21 更新で設計重心を「文脈ブロック一次成果物」へ再構成し、Open Questions を確定。実装指示ではないが、確定事項は実装の前提とする）。アーキテクチャ判断は [ADR-005](../adr/ADR-005-news-context-block-architecture.md) として確定（Accepted）。現状仕様の入口は [docs/wiki/news-context-blocks.md](../wiki/news-context-blocks.md)。

## 設計重心（2026-06-21 確定）

本機能の本質的価値は「記事生成」ではなく **「文脈ブロック生成」** にある。競合のテンプレ SEO サイトでも結果記事・速報記事は生成できるが、Softeni Pick が持つ大会・選手・試合データを横断して生成する文脈情報は再現が難しい。したがって**文脈ブロックを一次成果物として設計し、記事はその再利用先の一つ**として扱う。

採用するパイプライン:

```
大会データ
  ↓ イベント抽出（初優勝/連覇/王者撃破/通算節目/ベスト4初進出 など）
  ↓ 文脈ブロック生成（一次成果物）
  ↓ 再利用先：大会ページ / 選手ページ / ランキング / 記事
```

旧案の「大会データ → 記事生成 → 文脈ブロック」は採らない。記事を頂点に置くと、文脈が記事に従属して他面（大会・選手・ランキング）で再利用しにくくなるため。

### イベント抽出を上位概念に置く

文脈ブロックのさらに上位に **「イベント抽出」** レイヤを設ける。大会データから「初優勝」「連覇」「3連覇」「王者撃破」「通算50勝」「ベスト4初進出」などの**意味のある出来事を構造化データとして抽出**し、文脈ブロックはこのイベント列を入力に生成する。

- 利点: イベント列を一次データとして持てば、大会ページ・選手ページ・ランキング・記事の**すべてで再利用**できる。長期的に最も Softeni Pick らしい資産になる。
- 段階導入: 初期は具体ブロック（historical-winners / milestone / career-record）を直接実装するが、`milestone` は実質「イベント抽出」の最初の具体例。ブロック生成関数は、将来イベント列を介在させても破綻しない責務分割（抽出と整形を分ける）で書く。

## 背景と狙い

流入の大きい時事系（速報・プレビュー）は運用負荷が高く、現状の手動入力中心の運用では即応が難しい。一方で競合調査（後述）から、この領域は「速いが浅いテンプレ farm」と「深いが遅い編集メディア」に二極化しており、「速くて、かつ選手の履歴・文脈が自動で付く」枠が空いている。

Softeni Pick の差別化資産は `data/tournaments/details/**` を中心とした構造化データ（選手・チーム・大会・結果）であり、これは farm が原理的に持てず、編集メディアが人力で再現できない。したがって本機能の主眼は「速報の速さ」を競うことではなく、**確定済みデータに DB 由来の文脈を自動付与して、運用負荷を『チェックのみ』に圧縮する**ことに置く。

### 競合調査サマリー（2026-06-21）

- 汎用テンプレ SEO farm（zutto-sports.com / sposoku.com）: 全スポーツ横断のテンプレ量産。大会ページを空で先行作成→検索意図を先取り→結果を貼るだけ。文脈・分析はゼロ。「大会ページ事前作成→結果で埋める」手法は有効と実証されている。
- 編集・公式（softtennis-mag.com / JSTA 連盟公式 / 個人ブログ）: 本物の記事で深いが、人力依存でスケール・速度が出ない。
- 結論: 「{大会}{年}結果速報」キーワードで farm と正面勝負はしない（ドメインスケールで不利）。選手名・学校名 × 文脈のロングテールを狙い、既存の選手・歴代ページへ回遊させる。

## 前提（確定した設計判断）

ユーザーとの確認（2026-06-21）で次を採用する。

1. **データ取得**: 入力は完全手動のまま（PDF 抽出・`scripts/matches/**` 等の既存フロー維持）。**自動化するのは「生成」だけ**。外部速報元の自動クロールは行わない（保守・正確性・権利リスクを回避）。
2. **出力先**: **記事ページ新設 ＋ 既存ページ強化の両方**。同一の生成物（文脈ブロック）を記事ページと既存ページ（大会・選手）の双方で再利用する。
3. **生成エンジン**: **テンプレートのみ**。本文は DB 値のテンプレ埋め込みで決定的に生成し、LLM は使わない（誤り混入ゼロ・低コスト・鮮度シグナルの安定）。文章の自然さより、farm が出せない「DB 由来の事実の密度」で差別化する。
4. **公開フロー**: **human-in-the-loop**。自動でドラフト生成 → 人が承認 → 公開。承認なしに公開しない。

### この前提が意味すること（Assumption）

- 「速報」は厳密なリアルタイムではなく「手動入力が済んだ直後の半自動公開」になる。即時性では farm に劣るが、本機能の価値は速度ではなく文脈の自動付与とロングテール獲得にあるため、整合する。
- テンプレートのみのため、生成文の品質＝テンプレ設計と DB の網羅性に依存する。テンプレは「事実の列挙＋定型のつなぎ文」で構成する。

## 共通基盤（A・B 共有）

### データソース（既存・実体確認済み）

- `data/tournaments/details/<tournamentId>/<year>/<category>.json`
  - `participants`（id / lastName / firstName / team / prefecture）
  - `entries`（entryNo / playerIds / type: seed|packing 等）
  - `matches`（entries / scores / round / winnerEntryNo / retired / stage / group / matchId / nextMatchId / prevMatchIds）
  - `results`（entryNo / tournament.label / tournament.rank / roundrobin）
- `data/tournaments/information/<tournamentId>.json`（year / location / startDate / endDate / label / sourceUrl / categories）
- `data/tournaments/index.json` / `local_index.json`（大会一覧・世代・地域紐付け）
- `data/players/index.json`（id / lastName / firstName / count）
- `data/players/<slug>/analysis.json`（既存の自動生成。latestMatch などを保持）

これらは現行 source of truth（`docs/wiki/data-model.md` 準拠）。本機能は**新規データ源を増やさず**、上記の横断集計でほぼ賄える。

### 生成スクリプト（新規・仮称）

- `scripts/generate-news-drafts.mjs`
  - 役割: details / information / players を読み、記事ドラフト JSON を出力。
  - 既存 `prebuild` 系（`generate-players-json` → `generate-player-analysis` → `generate-beta-matches-json`）の後段に追加できる構成にする。
  - 出力: `data/news/drafts/<articleId>.json`（ドラフト置き場、git 管理）。
  - 決定的生成: 同一入力からは同一出力（差分レビューしやすく、鮮度シグナルを安定させる）。

### 記事エンティティ（新規データモデル）

`data/news/drafts/<articleId>.json` / 公開後 `public/data/news/<articleId>.json`（案）。

```
{
  "articleId": "ih-2025-doubles-boys-result",   // 安定 ID
  "type": "result" | "preview",                  // A or B
  "state": "draft" | "review" | "published",     // 承認ステートマシン
  "tournamentId": "...", "year": 2025, "category": "doubles-none-boys",
  "title": "...", "description": "...",           // テンプレ生成・一意化
  "publishedAt": null, "updatedAt": "...",        // 実データ由来（ビルド日を使わない）
  "blocks": [ /* 文脈ブロック配列。記事と既存ページで共用 */ ],
  "sources": [ /* 参照した details/information のパスや sourceUrl */ ]
}
```

### 承認ステートマシン

`draft` →（人が確認）→ `review` →（承認）→ `published`。

- 公開対象は `state === "published"` のみ。生成スクリプトは既存の `published` を勝手に `draft` へ戻さない（追記型。`generate-beta-matches-json.mjs` の追記型生成と同方針）。
- 承認 UI の置き場は Open Question（後述）。当面は JSON の `state` を手動で書き換える最小運用でも成立する。

### 文脈ブロック（テンプレ部品。本機能の一次成果物）

farm が持てない「DB 由来の文脈」を部品化し、大会ページ・選手ページ・記事の各面に差し込む。すべて `data/tournaments/details/**` 横断と `players/index.json` から決定的に算出する。

**価値とリスクは均一でない**ため、安全性・再利用価値で優先度を分けて段階導入する（2026-06-21 確定）。

#### 優先度A（先行実装）

- `historical-winners`: その大会の歴代優勝者一覧（過去年度 details の `results` の `{kind:"winner"}` から）。**最も安全**。過去年度結果から機械的に生成でき、誤判定リスクが低い。
- `milestone`: 初優勝 / 連覇 / 3連覇 / 初出場 等の節目（歴代＋通算から導出）。SEO だけでなく**選手ページ強化にも再利用**できる。実質「イベント抽出」の最初の具体例。
- `career-record`: 対象ペア/選手の通算成績（掲載大会内の勝敗・主要タイトル）。利用価値が高い。
  - **必須表記**: 「当サイト掲載大会分の通算」と明示する（実通算との誤認を防ぐ）。既存の選手ページ文言「収録試合は通算◯試合◯勝◯敗」（`results.tsx`）と表現を揃える。

#### 優先度B

- `seed-vs-result`: シード（`entries.type==="seed"`）と実際の到達ラウンド（`results.tournament.rank`）の対比（番狂わせ検出）。有益だが A 群より下。

#### 優先度C（当面導入しない）

- `head-to-head`: 対戦相手との過去対戦履歴（details 横断で同一ペア対戦を抽出）。**後回し**。同姓同名・名寄せ・ペア変更による誤判定リスクが高く、誤った対戦履歴は信頼性を大きく損なう。**名寄せ精度が十分に検証できるまで導入しない**。

名寄せ（姓名→player id 解決）は既存規約（`players/index.json` 参照、同姓同名は「最初の id」規約）に従う。A・B 群は集計対象が「優勝者」「到達ラウンド」など粒度が粗く名寄せ依存が小さいため先行可。head-to-head のみ名寄せ精度に強く依存するため C とする。

## 実装順序（2026-06-21 確定）

記事機能が未完成でも、既存ページの情報密度向上による SEO 効果を先に獲得する順序にする。**B（プレビュー）を A（結果速報）より先**に、**既存ページ強化を記事ページより先**に行う。

1. `historical-winners` 生成
2. `milestone` 生成
3. `career-record` 生成
4. 大会ページへの差し込み
5. 選手ページへの差し込み
6. preview 記事生成（B）
7. result 記事生成（A）
8. `head-to-head`（名寄せ精度の検証後）

### なぜ B（プレビュー）を A（結果）より先にするか

- SEO の需要曲線: 結果記事は「大会終了 → 需要急減」。プレビューは「大会前 → 需要発生 → 結果記事へ育成」。プレビューで先に検索面を取り、同一 URL で結果へ昇格させる方が需要を長く拾える。
- 品質リスク: 結果記事は優勝者・スコアの誤りが致命的。プレビューは注目選手・有力ペア中心でリスクが低い。

### なぜ既存ページ強化を記事より先にするか

記事ページ新設前に、大会ページへ「歴代優勝者・前回王者・優勝ペア通算成績」、選手ページへ「milestone・career-record」を差し込むだけで情報密度は大きく向上する。記事機能の完成を待たずに SEO 効果を取りに行ける。

> 注: 以下は A・B の詳細。**実装順序では B（プレビュー）が A（結果）より先**（Step 6 < Step 7）。記載順は機能説明の都合で A→B だが、着手順は上記「実装順序」に従う。

## A: 結果速報＋文脈自動ドラフト

### トリガー

大会結果データ（`details/<tournamentId>/<year>/<category>.json`）が手動入力で更新されたら、`generate-news-drafts.mjs --type result` を実行（手動コマンド or 既存ビルド後段）。完全手動入力前提のため、入力者が結果投入後にコマンドを叩く。

### 生成物

`type: "result"` の記事ドラフト。`blocks` は最低限:

1. 結果サマリー（優勝・準優勝・ベスト4、決勝スコア）← `results` + `matches`
2. `milestone`（連覇/初優勝など）
3. 優勝ペアの `career-record`（「当サイト掲載分」と明示）
4. `historical-winners` と今回の位置づけ
5. `seed-vs-result`（番狂わせがあれば）
6. `head-to-head`（**Step 8 以降**。名寄せ精度の検証が済むまで含めない）

### 出力先（両方）

- 記事ページ（`/news/<articleId>`。下記 SEO 参照）にドラフト。
- 既存の大会年度別ページ・該当選手ページへ同じ文脈ブロックを差し込み（記事公開と独立に、ブロック単位で再利用。実装順序では Step 4-5 で先行）。

## B: プレビュー自動生成（予約投稿）

### トリガー

`information.json` に**未来日程**の大会（`startDate` が未来）が登録され、かつ `entries`（出場選手/シード）が入った時点で `--type preview` を実行。大会日程は事前確定するため、farm の「空ページ先行作成」に相当する先回りを、中身のあるプレビューで行う。

### 生成物

`type: "preview"` の記事ドラフト。`blocks` は:

1. 大会概要（日程・会場・カテゴリ）← `information.json`
2. 注目選手 / 有力ペア（出場 `entries` × 各選手の `career-record`・直近成績）
3. 前回王者・`historical-winners`
4. シード勢の過去到達ラウンド

### 結果確定後の合流（重要）

プレビュー記事は**結果確定後に同一 URL で A（結果速報）へ差分更新**する。プレビューで先に検索面を獲得し、結果で内容を厚くして同じ URL を育てる。これは canonical 統一にも有利（URL を増やさず評価を集約）。

- 実装: `articleId` をプレビューと結果で共有し、`type` を `preview`→`result` に昇格、`blocks` を差し替える。

## SEO / カニバリ設計（`docs/wiki/seo.md` 運用ルール準拠）

新ページ種別の追加時は seo.md の重複マップに追記が必須。想定する重なりと制御:

- 速報/結果記事 × 大会年度別ページ（`/tournaments/.../[year]/.../[gender]`）
  - 重なり: 同一大会同一年の結果。
  - 制御（インテント分割）: 記事＝「確定直後の文脈・読み物」、年度別ページ＝「結果記録の一次データ」。記事の title は選手名 × 文脈ロングテール寄せ、年度別ページは従来通り「{大会}{年}結果」。記事→年度別/選手/歴代ページへ内部リンクで回遊。
  - 状態: 未判断（GSC 実測待ち）。farm と「{大会}{年}結果速報」で正面勝負しない方針を明記。
- プレビュー記事 × 大会ハブ（年なし）
  - 制御: プレビューは「{大会}{年} 注目選手・組み合わせ・展望」インテント。ハブは歴代まとめ。結果確定後はプレビュー URL を結果記事へ昇格（同一 URL 継続）。
- 文脈ブロックの既存ページ差し込み: 薄い選手ページ（`noindex` 選別済み、`PLAYER_INDEX_MIN_MATCHES` 未満）に差し込む場合、ブロック追加で実質コンテンツが増え index 復帰条件に影響しうる。差し込みが index 判定に与える影響は要確認。

### URL 設計（2026-06-21 確定）

**採用: `/news/<articleId>`（独立ツリー）。**

- 理由: 既存大会ページとのインテント分離がしやすい。
- 不採用: `/tournaments/.../preview` / `/tournaments/.../result`。既存大会ページとの距離が近すぎ、Google から見て「大会結果ページ」と「結果記事」の差異が曖昧になりやすい（カニバリ距離が近い）。
- 大会ページとの関連性は、記事→大会/選手/歴代ページへの内部リンクで担保する（URL ツリーの近接に頼らない）。

## Open Questions の解決（2026-06-21 確定）

当初の Open Questions を以下のとおり決定した。

1. **承認 UI の置き場** → 承認（human-in-the-loop）は**公開記事（`/news/` 新規公開面）にのみ適用**する。既存ページへのブロック差し込み（Step 4-5）は決定的生成のため**ビルド時自動**とし、別途の承認は挟まない（既存 `analysis.json` 自動生成と同格の信頼水準）。承認運用は当面、記事 JSON の `state` を手書きで `published` にする最小運用。記事機能（Step 6-7）の実装時に beta 管理面へ寄せる。
   - 根拠: 差し込むブロックは A 群（historical-winners / milestone / career-record）で誤判定リスクが低く、既存ページの自動生成物（選手 analysis）と同じ扱いにできる。新規公開面（記事）だけ人の承認を必須にする。
2. **記事 URL 種別** → `/news/<articleId>`（確定。上記「URL 設計」参照）。
3. **head-to-head の名寄せ精度** → 優先度C。名寄せ精度の検証が済むまで**導入しない**（Step 8）。A・B 群は粒度が粗く名寄せ依存が小さいため先行可。
4. **career-record の「掲載分通算」表記** → 「当サイト掲載大会分の通算」と明示。既存選手ページの「収録試合は通算◯試合◯勝◯敗」（`results.tsx`）と表現を揃える。
5. **ブロック差し込みの index 判定への影響** → **影響なし（確定）**。index 判定は `totalMatches = playerMatches.length`（試合数）に基づく（`src/pages/players/[id]/results.tsx` L785, L1049）。文脈ブロックは試合数を変えないため `PLAYER_INDEX_MIN_MATCHES` 判定に影響しない。閾値ロジックは現状維持。なお差し込みでコンテンツが厚くなること自体は「薄いページ」評価の改善に資する（プラス方向）。
6. **プレビュー生成タイミング** → 完全手動入力前提に合わせ**手動コマンド実行**。`entries` が details に入った大会のうち `information.startDate` が未来のものを対象に、入力者が `generate-news-drafts.mjs --type preview` を叩く。自動検知トリガーは作らない（運用方針と整合）。
7. **速報/結果記事 × 大会年度別ページのカニバリ実害** → `/news` 分離＋インテント分割で先行運用し、**GSC 実測後に judge（監視）**。farm と「{大会}{年}結果速報」で正面勝負しない方針は維持。

## 残 Open Questions（実装中に判断）

- `milestone` / イベント抽出の語彙確定（初優勝・連覇・3連覇・初出場・王者撃破・通算N勝・ベスト4初進出 などのキー定義と判定条件）。
- `head-to-head` 導入可否を判断するための名寄せ精度の検証方法（サンプル抽出・誤マッチ率の計測手順）。
- 記事 `articleId` の命名規約（プレビュー→結果で共有するため安定 ID が必要）。
- イベント抽出レイヤを「最初から薄く実装」するか「milestone を内包する形で後付け」するかの実装判断。
