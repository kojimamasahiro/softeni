# STリーグ ページ SEO ギャップ診断 ＋ 改善計画（2026-06）

戦略ブリーフ「STリーグ期間中の検索流入を最大化する」を、現状の実装に照らして診断し、優先度順の改善計画にまとめたもの。第4回（2026年12月想定）シーズンに向けた準備が前提。

> 位置づけ: これは discussion note（docs/raw）。実装に進むものは wiki / ADR へ反映する。

---

## 1. 結論サマリー

現状は「速報サイト」としての基礎は非常に良くできている。ページ構成（ハブ／年度／結果・順位／チーム／分析／歴代）、構造化データ、全タブの静的HTML出力、階層（Ⅰ/Ⅱ/Ⅲ）対応、intent分離まで踏まえており、土台は合格点。

一方で、ブリーフが狙う流入のうち **取りこぼしている層が3つ** ある:

1. **「○○ vs △△」系の対戦クエリ** — 個別対戦ページ（URL）が無く、ランキング対象にできていない。**最大の網羅ギャップ。**
2. **速報性の「見せ方」** — データ構造は速報対応だが、ユーザーへの鮮度提示（更新日時・直近結果・次戦）が無く、「ここが一番早い／全部ある」と感じさせる導線が弱い。
3. **回遊の取りこぼし** — チーム名・選手名がプレーンテキストで、`/teams/[teamId]`・`/players/[id]` に繋がっていない。「チーム名」検索からの相互流入が断たれている。

第4回（2026）の実データ用ディレクトリがまだ無い（editions.json にサンプル記載のみ）点も、シーズン前の必須準備。

---

## 2. 現状の強み（維持すべき資産）

- **ページセットが網羅的**: `/st-league`（ハブ）／`/[year]`（年度トップ）／`/[year]/matches`（結果・順位）／`/teams`／`/analysis`／`/champions`／`/about`。
- **intent分離が効いている**: 年度トップ＝概要＋導線、matches＝順位表・対戦結果、とカニバリ回避済み。
- **構造化データ**: ItemList / SportsEvent / BreadcrumbList を各所で出力。
- **全タブを静的HTMLに出力**: 男女 × Ⅰ/Ⅱ/Ⅲ × プレーオフ を `hidden` 切替で全部HTMLに含めており、非アクティブタブもクローラブル。これは多くの競合が落とすポイントで、明確な優位。
- **データ蓄積の骨格**: editions.json（歴代王者）＋ promotionRelegation（昇降格の系譜）＋ champions ページ（常緑）。
- **パンくず・サブページ相互リンク**が一通り存在。
- **ドキュメント規律**（wiki / ADR / open-questions）が整っている。

## 3. ギャップ診断（戦略4優先度ごと）

### 優先度① 検索需要ページの網羅

| 項目 | 現状 | 評価 |
| --- | --- | --- |
| トップ / 年度 / 順位 / 試合結果 / チーム | あり | ◎ |
| **対戦カードページ（○○ vs △△）** | **無し**。対戦は matches 内の展開行で、個別URLが無い | **✕ 最大ギャップ** |
| チーム別の対戦成績・通算 | 無し（年度内のみ） | △ |
| title/description の年号最適化 | matches の title が「第N回STリーグ」主体。ユーザーは「STリーグ 2025」で検索する比率が高い | △ |

「STリーグ 結果」「STリーグ 順位」はハブ／matches で受けられる。だが**ブリーフが明示する「○○ vs △△」「対戦カード」を受けるURLが存在しない**。総当たり戦は対戦数が多く（Ⅰ部8チーム＝28対戦×男女＝56＋）、各対戦に固有ページを与えればロングテール面が一気に増える。

### 優先度② シーズン中の更新速度・速報性

| 項目 | 現状 | 評価 |
| --- | --- | --- |
| データ構造の速報対応 | match に `status`(finished/予定) あり | ◎ |
| **更新日時の表示 / dateModified** | **無し**。ユーザーもクローラーも鮮度を判断できない | ✕ |
| **直近結果・本日の試合・次戦の強調** | **無し**。最新結果がページ上部に出ない | ✕ |
| 第4回(2026)実データの受け皿 | **未作成**（サンプルのみ） | ✕ 要準備 |
| 反映フロー | `output: export` の静的書き出し。更新＝再ビルド＆デプロイ | △ 速度依存 |

「試合終了後すぐ更新」を掲げるなら、(a) 更新の鮮度をユーザーに見せる、(b) 入力→公開のリードタイムを短くする、の両輪が要る。今は (a) がゼロ。

### 優先度③ 「次に知りたい情報」への回遊

| 項目 | 現状 | 評価 |
| --- | --- | --- |
| 年度トップ ⇔ サブページ | 相互リンクあり | ◎ |
| **チーム名 → `/teams/[teamId]`** | **未リンク**（teams/matches ともプレーンテキスト） | ✕ |
| **選手名 → `/players/[id]`** | **未リンク** | ✕ |
| `/teams`・`/players` → STリーグへの逆流入 | 弱い（データ参照のみ） | △ |
| 過去対戦・前回大会への導線 | 仕様書5.5に記載あるが未実装 | △ |

ブリーフの「チームページへ」「次戦へ」「過去対戦へ」という回遊設計のうち、**チーム・選手への横移動が断線**している。検索ユーザーの「次に知りたい」の筆頭（このチーム/選手をもっと見たい）を取りこぼしている。

### 優先度④ データの資産化（常緑・翌年SEO）

| 項目 | 現状 | 評価 |
| --- | --- | --- |
| 歴代優勝 | champions にあり | ◎ |
| 昇降格の系譜 | あり | ◎ |
| **歴代順位表（部別・年度横断）** | 無し | △ |
| **チーム別通算成績 / 対戦成績** | 無し | △ |
| Ⅱ部女子・Ⅲ部データ | 方針として未保持（既知・文書化済） | 設計判断 |

常緑の核（歴代王者）は押さえている。あと一段、**チーム軸の通算データ**を積むと「STリーグを調べるならまずここ」の決定打になり、シーズン外流入と被リンク獲得に効く。

## 4. 改善計画（優先度順）

効果（検索流入インパクト）× 工数 で並べた。P0=今季前に必須、P1=今季中に効く、P2=常緑の積み増し。

### P0 — 第4回シーズン前に必須（受け皿と速報性の最低ライン）

1. **2026 実データ用ディレクトリの確定**
   `data/st-league/2026/{league,participants,matches}.json` を実データで用意（サンプルを差し替え）。出場確定し次第 participants → 日程確定で matches(status:予定) を先行投入し、**開幕前から「日程」クエリで露出**させる。
   - 効果: 大 / 工数: 小（データ入力主体）

2. **更新日時（最終更新）の表示 ＋ `dateModified` 構造化データ**
   各ページに「最終更新: YYYY/MM/DD HH:mm」を出し、SportsEvent/WebPage に `dateModified` を付与。鮮度をユーザーとクローラー双方に伝える。
   - 効果: 中〜大（速報性の根幹）/ 工数: 小

3. **更新→公開のリードタイム短縮の確認**
   静的書き出しのため、試合終了→反映に再ビルド＆デプロイが必要。シーズン中の運用で許容できる時間か検証し、必要なら差分ビルド or 該当ページのみのISR的運用を検討（別ADR候補）。
   - 効果: 中 / 工数: 中（要調査）

### P1 — 今季の流入を最大化する本丸

4. **個別対戦ページ `/st-league/[year]/matches/[matchId]`（○○ vs △△）**
   matches の展開行を独立URL化。title「{チームA} vs {チームB} 結果｜第N回STリーグ {年}」、D1/S/D2の個別スコア、両チーム順位・次戦・逆カード（過去対戦）へのリンクを内包。SportsEvent(個別試合) 構造化データ。
   - 既存 matches ページからは各対戦行に「詳細 →」リンクを足し、展開UIは残す（回遊と網羅の両取り）。
   - 効果: **大**（ロングテール面の最大化＋「vs」クエリ獲得）/ 工数: 中

5. **チーム名・選手名のリンク化（回遊の断線解消）**
   matches / teams / analysis のチーム名 → `/teams/[teamId]`、選手名 → `/players/[id]` をリンク化。teamId は participants.json に既にある（例 `ntt-west`）ので接続容易。逆に `/teams/[teamId]` 側にも「STリーグでの成績」セクションを足し相互リンク。
   - 効果: 大（内部リンク密度↑＝順位安定＋回遊）/ 工数: 中

6. **「直近の結果・本日／次の試合」ブロックを matches 上部に追加**
   開催中は最新の finished 数件と、直近の予定（次戦）を上部に固定表示。「ここを見れば最新が分かる」体験を作る。
   - 効果: 中〜大 / 工数: 小〜中

7. **title/description の年号最適化**
   matches/年度トップの title に西暦年を明示（例「STリーグ2025 結果・順位表（第3回）」）。「STリーグ {年}」の検索意図に合わせる。
   - 効果: 中 / 工数: 小

### P2 — 常緑データの積み増し（翌年以降に効く）

8. **チーム別ページ `/st-league/teams/[teamId]`（または既存チームページ内セクション）**
   STリーグ出場各社の**通算成績・年度別順位・主要対戦の履歴**を集約。被リンク・指名検索の受け皿。
   - 効果: 中（常緑・翌年SEO）/ 工数: 中〜大

9. **歴代順位表（部別・年度横断）を champions に追加**
   王者だけでなく各回の最終順位を一覧化。「STリーグ 歴代 順位」を取りに行く。
   - 効果: 中 / 工数: 小〜中

10. **about に FAQPage 構造化データ**
    「STリーグとは」「Ⅰ・Ⅱ・Ⅲの違い」「入替戦とは」をFAQ化しリッチリザルト狙い。
    - 効果: 小〜中 / 工数: 小

## 5. 推奨ロードマップ

- **〜開幕前（最優先）**: P0-1（データ受け皿）→ P0-2（更新日時）→ P1-7（title年号）。日程確定で予定matchを先行公開。
- **開幕直前〜会期中**: P1-4（対戦ページ）と P1-5（リンク化）を投入。P1-6（直近・次戦）で速報体験を仕上げる。会期中はP0-3の運用検証を回す。
- **閉幕後〜オフ**: P2-8/9/10 で常緑データを積み、翌年の土台にする。

一言で: **「対戦ページ（網羅）＋リンク化（回遊）＋更新日時/直近結果（速報の見せ方）」の3点が、今季の流入を一段引き上げる本丸。** データ受け皿の準備だけは開幕前に必ず先行させる。

---

## 6. 決定事項（2026-06、ブリーフ協議で確定）

- **個別対戦ページの粒度**: **Ⅰ部の決着済み対戦のみ**を個別URL化（`/st-league/[year]/matches/[matchId]`）。Ⅱ部・プレーオフ・予定試合は matches ページ内の展開行のまま（URL無し）。薄ページ・インデックス膨張を回避し、需要の濃いⅠ部「vs」クエリに集中。効果を見てから拡張を判断。
- **速報運用**: まずは**手入力**で運用開始。会期中に出た課題（リードタイム等）を都度潰す方針。半自動取り込みは将来検討。
- **チーム軸の通算データ**: 専用ページは作らず、**既存 `/teams/[teamId]` 内の「STリーグでの成績」セクション**として実装（通算W-L・年度別順位・主要対戦履歴）。`/teams` ⇄ STリーグの相互リンクで回遊を強化。
- **Ⅱ部女子データ**: 公式PDFを入手して解析・入力する（ユーザーよりPDF提供予定）。入力後 `hasMatchData:true` に切替え「準備中」表示を解除。

> これらが Accepted 化したら wiki（`docs/wiki/st-league.md`）へ反映。対戦ページ新設は public URL 契約の変更にあたるため ADR 追加を検討。
