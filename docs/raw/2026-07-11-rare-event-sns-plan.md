状態: Draft（詳細検討。親: [2026-07-11-idea-rare-point-event-sns.md](./2026-07-11-idea-rare-point-event-sns.md)）

## 1. 誰に刺さるか（ターゲット分析）

この案の拡散は「コンテンツの面白さ」より「**自分ごと化**」で起きる、と置く。

| 層 | 刺さる理由 | 期待する行動 |
|---|---|---|
| **当事者圏**（出場選手・チームメイト・保護者・同校OB） | 「自分/うちの子/後輩が映っている」。競技人口の65%が中学生＝拡散の実行者は本人でなく周囲（保護者・部活仲間） | RT/シェア・保存。投稿の初速を作る |
| **競技ファン**（大会結果を追う既存読者） | 「結果は知ってるが、その瞬間は見ていない」。動画タイムスタンプが結果テキストに"瞬間"を足す | サイト回遊・リピート閲覧 |
| **休眠層・競技外**（中→高で離脱した経験者、他競技ファン） | 数字付きの驚き（「21球ラリー」「大会最長」）は競技知識ゼロでも通じる | 認知。普及エンジンとしての本来目的 |

主対象は**当事者圏**。初速とエンゲージメントを作れるのはここだけで、競技外への到達は当事者圏の拡散の結果として起きる（ローカルスポーツSNSの一般的な構造）。したがって投稿設計は「本人・保護者が誇らしくシェアできるか」を第一基準にする。

## 2. 多くのユーザーに刺さるための投稿設計

### 2.1 カテゴリは「相対評価＋ポジティブ限定」から始める

パイロットチェック（16試合・717ポイント）の教訓をそのまま設計に反映する。

- **絶対閾値は使わない**（逆転勝ち11%・完封13.8%は「珍しくない」上に投稿過多で特別感が死ぬ）。
- **大会単位の相対評価**を既定にする: 「この大会で最長のラリー」「この大会唯一のサービスエース」は、どの大会でも**必ず1件は成立**する（歩留まり問題の構造的解決）。
- **ポジティブに語れるものだけ自動候補化**する。完封・連続失点など片方を下げるカテゴリは、当事者圏（多くは未成年）が主対象である以上、初期スコープから外す（人手レビュー前提でも扱わない方が安全。将来「守備側の粘り」等ポジティブ変換できたら再考）。

初期カテゴリ案（すべて `superlative`＝大会内最上級で判定）:

| カテゴリ | 判定 | データ根拠 |
|---|---|---|
| 大会最長ラリー | `rally_count` 最大のポイント | 21球以上は0.6%と実際に希少 |
| サービスエース | `result_type: service_ace` 全件 | 0.3%と実際に希少なので絶対判定でも成立 |
| 大会最長デュース | ゲーム内ポイント数最大（matchRules で再構成） | デュース自体は26.6%だが「最長」は1件 |
| 最大ビハインドからの逆転ゲーム | 落差最大の1件のみ | 逆転自体は11%だが「最大」は1件 |
| 大会最多連続ポイント | 同一サイド連取の最長 | 要頻度確認 |

### 2.2 投稿フォーマット（テンプレート主義）

ADR-005 の「本文はLLMでなくテンプレート」方針をSNSにもそのまま適用する。

```
🏸 {大会名} この大会最長の{N}球ラリー
{選手A}/{選手B} vs {選手C}/{選手D}（{ラウンド}）
その瞬間はこちら👇
https://youtube.com/watch?v={id}&t={video_start_ms/1000}s
{試合詳細ページURL}
```

- 動画はクリップ再配布でなく**タイムスタンプ付きURL**（著作権リスク低・実装コスト小。既定方針を維持）。
- 末尾に必ず自サイトの試合詳細URLを添え、SNS→サイト回遊の導線にする（siteLink 済みのネストURL）。
- 実名の扱い: **大会結果としてすでに実名公開している範囲と同一**に限定（運営記録のメジャー大会のみ）。ADR-003/004 のUGC/成長分析プライバシー論点に踏み込まない、という既定方針を投稿テンプレにも貫く（成績・希少プレー以外の評価的言及はしない）。

### 2.3 プラットフォーム

X（旧Twitter）を第一候補とする（Assumption: ソフトテニス界隈の大会速報・部活アカウントの主要生息地。運用開始後に Instagram Reels 等への展開を再検討）。投稿は当面**手動**（レビューを兼ねる）。

## 3. 実装構成

### 3.1 データフロー（ADR-005 のイベント抽出レイヤに乗せる）

```
score機能で大会を手動記録（match_video_sessions / video_start_ms 付き）
  ↓ public/data/beta-matches/matches/*.json（既存生成物）
  ↓ lib/rarePointEvents.ts（新規・純関数）: 大会単位でポイント列を横断集計
  ↓ RarePointEvent[]（構造化イベント。milestone と同じ思想＝文章でなくイベント列が一次成果物）
  ↓ scripts/generate-rare-point-events.mjs: 候補一覧＋投稿テンプレ文＋URLを出力
  ↓ 人がレビューして投稿（P0は手動コピペ）
  ↓ 将来: /news 記事の文脈ブロック（blockType: 'rare-point-events'）として再利用
```

### 3.2 スキーマ案

```ts
type RarePointEventKind =
  | 'longest-rally'        // 大会最長ラリー
  | 'service-ace'          // サービスエース（全件）
  | 'longest-deuce'        // 大会最長デュースゲーム
  | 'biggest-comeback'     // 最大ビハインド逆転ゲーム
  | 'longest-point-streak';// 最多連続ポイント

type RarePointEvent = {
  kind: RarePointEventKind;
  scope: 'tournament';             // 将来 'season' | 'all-time' へ拡張
  tournamentId: string; year: number;
  matchId: string; gameId?: string; pointId?: string;
  players: { display: string; team: 'A' | 'B' }[];
  detail: Record<string, string | number>;  // rallyCount, deficit, streak 等
  videoUrl?: string;               // タイムスタンプ付きYouTube URL（video_start_ms がある場合のみ）
  postable: boolean;               // videoUrl 有無＋ポジティブ判定
  label: string;                   // テンプレ素文（milestone と同規約）
};
```

- スコア再構成は `lib/matchRules.ts` を再利用（パイロット分析と同じ手法）。
- `video_start_ms` が無いポイントは `postable: false`（サイト内表示のみ）。動画リンクの有無が投稿価値を大きく左右するため、記録運用側で「候補になりそうなポイントは動画レビューを優先する」運用と対にする。

### 3.3 フェーズ分け

| Phase | 内容 | 完了条件 |
|---|---|---|
| **P0**（次の記録大会で） | `lib/rarePointEvents.ts`＋生成スクリプト。出力はコンソール/JSONの候補一覧。投稿は手動 | 1大会ぶん記録→候補が1件以上出る→実際に1投稿する |
| **P1** | 投稿実績を見てカテゴリ調整。beta 管理面に候補レビューUI（既存 beta 面の流儀に合わせ noindex） | 投稿までの作業が「レビュー＋ワンクリックコピー」に短縮 |
| **P2** | ADR-005 文脈ブロック化（大会ページ・/news 記事に「この大会の名場面」欄）。milestone 系（姉妹案）と発信キューを統合 | 記事/大会ページに自動差し込み |
| **P3**（反響次第） | scope 拡張（'season' / 'all-time': 「今季最長」「サイト記録更新」）。選手ページへの蓄積＝「その選手の物語」化 | — |

### 3.4 KPI（P0-P1 で見るもの）

- 大会あたり投稿可能候補数（目標: 3件以上/大会。相対評価設計の検証）
- 投稿エンゲージメント（当事者圏に届いたかは RT 元アカウントの属性で定性確認）
- 投稿→試合詳細ページへの流入

## 4. 課題・未解決（更新）

- 記録労力は残る（本案は「記録した大会の歩留まりを最大化する」ものであり、記録自体は減らない）。1大会の記録コスト実測を P0 で取る。
- `rally_count` の記録品質（basic 記録で 0 のポイントが存在）。集計時は `rally_count > 0` のポイントのみを分母にする等のガードが要る。
- X の API/自動投稿は P1 以降も当面不要（手動投稿で十分な頻度）。自動化判断は投稿頻度が週2を超えてから。
- 大会主催者・配信者への仁義（タイムスタンプリンクは法的リスクが低くても、配信元への事前一声が長期的に安全）。
