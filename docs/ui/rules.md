# rules.md — プロジェクト運用ルール

- ステータス: approved(Phase 0 成果物)
- すべての作業(人間・AI問わず)はこのルールに従う。担当AIが変わってもルールは変わらない。

## フェーズ運用

- **R1. フェーズを飛ばさない。** フェーズは Phase 0→8 の順に直列で進める。並行実施も禁止。
- **R2. レビュー前に次へ進まない。** 成果物を `in-review` にしてオーナーへレビュー依頼し、承認を得るまで次フェーズの作業を開始しない。
- **R3. 承認された内容のみを次フェーズで利用する。** 次フェーズの入力にできるのは `approved` の成果物だけ。draft を前提に作業しない。
- **R4. 1セッションの作業範囲を宣言する。** セッション開始時に「今回どこまでやるか」をオーナーに一言宣言してから着手する。

## 記録・状態管理

- **R5. 変更履歴を残す。** project-status.md の更新履歴、および各成果物の改訂は必ず記録する。承認済み成果物を黙って書き換えない。
- **R6. 承認はオーナーの明示的な発言のみ有効。** AIが自己判断で `approved` にすることを禁止。承認は decisions.md に日付つきで記録する。
- **R7. 決定事項・未決事項を即時管理する。** 会話中に決まったことはその場で decisions.md へ、決まらなかったことは project-status.md の未決事項へ記録する。会話ログにしか残っていない決定は「決定していない」とみなす。
- **R8. ドキュメントを常に最新へ。** 作業単位が終わるたびに関連ドキュメントと project-status.md を更新する。セッション終了時の更新は必須(次回開始位置を含む)。
- **R9. 情報の置き場所を守る。** 進捗は project-status.md、決定は decisions.md、設計は deliverables/、ルールは rules.md。同じ情報を二重に書かない。

## 品質・スコープ

- **R10. 事実と提案を分離する。** 調査結果(事実)と改善案(提案)を同じ節に混ぜない。特に Phase 1 は事実のみ。
- **R11. レビューしやすく出す。** レビュー依頼には必ず「見てほしい観点」を3〜5個添える。大きな成果物は節ごとに分割してレビュー依頼してよい。
- **R12. 差し戻しは正式手続きで。** 後続フェーズで前フェーズの前提が崩れた場合、黙って修正せず、(1) 矛盾を project-status.md に記録 → (2) オーナーに差し戻しを提案 → (3) 承認後に前フェーズ成果物を `in-review` に戻して改訂する。
- **R13. スコープ外に踏み込まない。** 各フェーズの「実施内容」に書かれていない作業(先回りの実装・デザイン)は行わない。気づいた点は未決事項として記録するに留める。
- **R14. 用語は glossary.md に従う。** 未定義の用語を使う必要が生じたら、まず glossary.md に定義を追加(draft)してから使う。
- **R15. 進行中のサイト変更は割り込ませない。** プロジェクト期間中にサイト本体へ機能追加があった場合、未決事項に登録し、現行フェーズ完了時に影響をまとめて反映する。
