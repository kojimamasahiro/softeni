# Softeni Pick UI/情報設計 改善プロジェクト計画書

- ステータス: approved(Phase 0 成果物)
- 作成日: 2026-07-03
- 本書はプロジェクト全体の計画を定義する。進捗は本書ではなく `project-status.md` で管理する。

---

## 1. プロジェクト全体像

### 1.1 目的

Softeni Pick(ソフトテニスのデータサイト)の情報構造・UI・コンポーネントを体系的に整理し、
**今後データや機能が増え続けても一貫性を保ったまま拡張できる設計基盤と運用ルールを確立する。**

### 1.2 ゴール

- サイト全体の情報構造・ページ構造・コンポーネントが文書化され、承認済みである
- 新しいページ・機能を追加するとき、`docs/ui/` を参照するだけで設計判断ができる
- 誰(どのAIエージェント)が作業しても同じ設計判断に到達できる

### 1.3 完了条件

以下がすべて満たされたときプロジェクト完了とする。

1. Phase 1〜8 のすべての成果物が「承認済み(approved)」である
2. 移行計画(08-migration-plan.md)がレビュー・承認済みである
3. 運用ガイド(09-operation-guide.md 相当、Phase 8 成果物)が承認済みである
4. `project-status.md` の現在フェーズが「完了」になっている

### 1.4 成功基準

| # | 基準 | 検証方法 |
|---|------|----------|
| S1 | 新規ページを追加する際、既存ドキュメントのページタイプ・コンポーネントで設計判断が完結する | 実際に架空の新機能でウォークスルーする |
| S2 | 新しいAIエージェントが PROJECT.md → project-status.md を読むだけで、1セッション内に作業を再開できる | フェーズ切替時に毎回検証される |
| S3 | 用語・情報構造の定義が一意(同じ概念に複数の名前がない) | glossary.md との突合 |
| S4 | 決定事項がすべて decisions.md に記録され、理由を遡れる | レビュー時に確認 |

### Phase 1: 現状調査(インベントリ)

| 項目 | 内容 |
|------|------|
| 目的 | 現状を**事実として**網羅的に棚卸しする(評価・改善案は含めない) |
| 実施内容 | 全ページの一覧化(URL・目的・主要データ)/ ナビゲーション構造の記録 / 使用中のUI要素・表示パターンの列挙 / データ種別(選手・試合・大会・ランキング等)の列挙 |
| 入力 | 現行サイト・ソースコード・オーナーからの補足情報 |
| 成果物 | deliverables/01-inventory.md |
| レビュー項目 | ページの網羅性(漏れがないか)/ 事実と評価が分離されているか / データ種別の粒度が適切か |
| 完了条件 | オーナーが「現状はこの通り」と承認する |
| 次フェーズへ進む条件 | インベントリが approved になり、未調査項目が未決事項として記録されている |

## 3. ドキュメント構成

### 3.1 ディレクトリ構成(docs/ui/ 配下)

```
docs/ui/
├── PROJECT.md              # 入口。新しいAIは必ず最初に読む
├── project-status.md       # 状態管理(進捗の単一情報源)
├── project-plan.md         # 本書(計画・フェーズ定義)
├── rules.md                # 運用ルール
├── decisions.md            # 決定事項ログ(理由つき)
├── glossary.md             # 用語集
├── deliverables/           # フェーズ成果物(資産として残る)
│   ├── 01-inventory.md
│   ├── 02-issues.md
│   ├── 03-domain-model.md
│   ├── 04-site-structure.md
│   ├── 05-page-structure.md
│   ├── 06-design-principles.md
│   ├── 07-components.md
│   ├── 08-migration-plan.md
│   └── 09-operation-guide.md
├── reports/                # フェーズ完了報告(履歴として残す)
│   ├── phase-1-report.md
│   └── ...
└── templates/
    └── phase-report.md     # フェーズ完了テンプレート
```

### 3.2 命名規則

1. すべて小文字の kebab-case(例: `site-structure.md`)。例外は入口の `PROJECT.md` のみ(視認性のため大文字)
2. フェーズ成果物は 2桁のフェーズ順プレフィックス(`01-` 〜 `09-`)。読む順=番号順になる
3. フェーズ報告は `phase-{N}-report.md`
4. 日付はすべて `YYYY-MM-DD`
5. ドキュメント冒頭に必ずステータスを記載: `draft` → `in-review` → `approved`
6. 承認済みドキュメントを改訂する場合はステータスを `in-review` に戻し、decisions.md に変更理由を記録する

### 3.3 各ドキュメントの役割の分離(重要)

- **進捗・状態** → project-status.md のみに書く(他に書かない)
- **決定と理由** → decisions.md のみに書く
- **設計内容** → deliverables/ のみに書く
- **ルール** → rules.md のみに書く

同じ情報を複数の場所に書かないこと。二重管理は引き継ぎ事故の最大要因。

## 8. この計画の懸念点と改善案

| # | 懸念 | 改善案 |
|---|------|--------|
| 1 | フェーズが8つと長く、直列のため完了まで時間がかかる | 各フェーズの成果物を「レビュー可能な最小単位」で区切る。1フェーズ内でも節ごとにレビュー依頼してよい(rules.md R11) |
| 2 | Phase 1 の調査が発散しやすい(どこまで細かく調べるか) | 着手前にインベントリの記載テンプレート(列の定義)をオーナーと合意してから調査を開始する |
| 3 | レビューがオーナー1人に集中しボトルネック化する | フェーズ報告に必ず「レビュー観点チェックリスト」を付け、見るべき箇所を絞って提示する(templates/phase-report.md に組込済) |
| 4 | プロジェクト進行中にサイトへ機能追加が入り、インベントリが古くなる | 変更受付ルール: 進行中の機能追加は project-status.md の未決事項へ登録し、現行フェーズは止めない。影響はフェーズ完了時にまとめて反映 |
| 5 | 引き継ぎ時、全ドキュメントを読むとコンテキストを圧迫する | PROJECT.md に「現在フェーズで読むべきファイルだけ」を判定できる読書順を定義。過去フェーズ成果物は必要時のみ参照 |
| 6 | 設計が進むほど初期フェーズの成果物と矛盾が生じる可能性 | 後続フェーズで前提が崩れた場合は「差し戻し」を正式な手続きとして定義(rules.md R12)。黙って上書きしない |
| 7 | 「承認」の基準が曖昧になりがち | 承認はオーナーの明示的な発言のみ有効。AIの自己判断による approved 化を禁止(rules.md R6) |
