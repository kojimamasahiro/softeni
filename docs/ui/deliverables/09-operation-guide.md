# 09-operation-guide.md — 運用ガイド

- ステータス: approved(D-010, 2026-07-04)
- 作成日: 2026-07-04
- フェーズ: Phase 8
- 役割: プロジェクト完了後、新ページ・新機能を追加するときの判断手順。**このガイドと docs/ui/ だけで設計判断が完結する**ことを目的とする(成功基準 S1)

## 1. 新ページを追加するときの手順

1. **エンティティ確認**(03-domain-model.md): 扱う情報は既存エンティティか? 新概念なら glossary.md に draft 追加 → 03 の改訂を提案
2. **URL 決定**(04-site-structure.md §5): 命名規則に従い、サイトマップ上の位置を決める。`/beta` に公開コンテンツを置かない
3. **ページタイプ割当**(05-page-structure.md §3): T1〜T7 のどれかに割り当てる。割り当てられない場合は保留として project-status.md に記録し、オーナーと相談(安易に新タイプを作らない)
4. **構成要素**(07-components.md §5): タイプ別の必須コンポーネントを使う。選手名・チーム名は必ず PlayerLink / TeamLink
5. **見た目**(06-design-principles.md): トークンのみ使用。新しい色・余白が必要なら 06 の改訂を提案
6. **公開整備**: パンくず+MetaHead+JSON-LD、sitemap・llms.txt への反映、ナビ導線(グローバルはセクション入口のみ・サブナビ活用)
7. **記録**: 04 の対応表(§6)へ行を追加

## 2. 新機能・新カテゴリ(特集など)を追加するとき

- 特集の追加基準: 04 §8(独自データまたは独立した検索需要がある場合のみ)
- 一覧 → ハブ → 詳細の3層を基本形にする
- 大会データは必ず `/tournaments/[generation]/**` に置き、特集はその切り出しとする

## 3. コンポーネントを追加・変更するとき

- 07 §6 の基準(2ページ以上で共通化、責務1文、依存方向 feature→data→base)
- 統廃合表(07 §2)への登録を忘れない

## 4. ドキュメント更新ルール

| 変更内容 | 更新するファイル |
|----------|----------------|
| ページ追加・URL 変更 | 04(対応表)、public/llms.txt、sitemap 設定 |
| 用語追加 | glossary.md(draft)→ 03 改訂 |
| トークン追加・変更 | 06(オーナー承認事項) |
| コンポーネント追加・統廃合 | 07 |
| 設計判断・方針決定 | decisions.md(理由つき・即時) |
| 承認済みドキュメントの改訂 | ステータスを in-review へ戻し、decisions.md に理由を記録(rules.md R5/R12) |

- 二重管理禁止(進捗=project-status.md、決定=decisions.md、設計=deliverables/)は運用後も維持する

## 5. 成功基準のウォークスルー(S1 検証)

**架空の新機能「中学(junior-highschool)特集」を追加する場合:**

1. エンティティ: 世代 `junior` は既存。「中学チーム」は高校チームと同型のサブセット → glossary に draft 追加
2. URL: 特集は `/junior-highschool` をトップレベルに(04 §5-6 の拡張ルール)。大会データは既存 `/tournaments/junior/**` のまま
3. ページタイプ: 特集トップ=T2、都道府県=T3、学校=T3、歴代記録=T4(高校特集と同型)
4. コンポーネント: SegmentedControl(男女)、ResultsTable、TeamLink を再利用。新規実装は原則不要
5. ナビ: 「特集」グループに「中学」を追加(グループ7項目以内)

→ 既存ドキュメントのみで全設計判断が完結することを確認(S1 充足)。

## 6. プロジェクト完了後の本ドキュメント群の位置づけ

- docs/ui/ は「設計の正」として維持する。docs/wiki/ (実装仕様)と役割が異なる: wiki=実装の現状記録、ui=設計判断の基準
- 矛盾が生じた場合は decisions.md に記録の上、docs/ui/ を優先して wiki を追従させる
