# AdSense 広告 UI 設計提案

Status: Proposal(未実装)
Date: 2026-06-12

## 現状と課題

- `src/pages/_document.tsx` でスクリプト読込のみ。手動広告ユニットなし、完全に自動広告任せ。
- 課題:
  - 自動広告は挿入位置を制御できず、テーブル途中やファーストビューに割り込み UX が崩れる。
  - 挿入時に高さ確保がなくレイアウトシフト(CLS)が発生 → Core Web Vitals 悪化は SEO・収益双方に悪影響。
  - 収益の柱になる「コンテンツに沿った固定枠」がない。

## 方針: 手動枠 + 自動広告の併用

収益の柱となる枠は手動で固定配置し(位置と高さを制御)、自動広告はアンカー広告など補完用途に絞る。

## 1. AdUnit コンポーネント設計

`src/components/AdUnit.tsx` を新設(実装時)。要点:

- **高さの事前確保**: ラッパー div に `min-height`(モバイル 280px / レスポンシブ枠)を指定し CLS をゼロにする。広告が埋まらない場合も空間は維持(潰すと逆にシフトする)。
- **「スポンサーリンク」ラベル**: 上部に小さく表示。誤クリック防止とポリシー準拠。
- **SPA 遷移対応**: Next.js のルート遷移ごとに `(window.adsbygoogle = window.adsbygoogle || []).push({})` を再実行(`useEffect` + `router.asPath` キー)。これを怠ると 2 ページ目以降広告が出ない。
- **ダークモード**: ラッパー背景は `bg-gray-50 dark:bg-gray-800` 程度の控えめな色で「広告予定地」が悪目立ちしないようにする。
- **スロット ID は props で受ける**: ページ種別ごとに AdSense 管理画面でユニットを分け、収益をユニット単位で比較できるようにする。

## 2. 配置設計(優先ページ)

原則: **ファーストビュー(h1 と最初のコンテンツ)には置かない**。ユーザーが目的を達成した「区切り」の直後に置く。

### players/[id](選手詳細)

| 位置                                    | 枠                       |
| --------------------------------------- | ------------------------ |
| プロフィール section 直後・試合結果の前 | レスポンシブ枠 ①(主力枠) |
| 試合結果 section 直後・関連リンクの前   | レスポンシブ枠 ②         |

プロフィールを見終えた区切りが最も視認性が高く、テーブル(試合結果)を分断しない。

### teams/[teamId](チーム詳細)

| 位置                                        | 枠                 |
| ------------------------------------------- | ------------------ |
| 紹介文 section 直後・年度別成績の前         | レスポンシブ枠 ①   |
| 年度ブロックの間(2 年度ごとに 1 枠、最大 2) | インフィード風枠 ② |

### matches/[matchId](試合詳細)

実体は `beta/matches-results/[matchId]` で section カードが縦に並ぶ長尺ページ。

| 位置                             | 枠                       |
| -------------------------------- | ------------------------ |
| スコアボード(最初のカード)直後   | レスポンシブ枠 ①(主力枠) |
| 統計/分析カードの間(中間 1 箇所) | レスポンシブ枠 ②         |
| ページ末尾・フッター前           | レスポンシブ枠 ③(任意)   |

カードとカードの間に同じ角丸スタイルのラッパーで置くと UI に馴染む。カード内部・テーブル内部には入れない。

### 共通ルール

- 1 ビューポート内に広告 2 枠以上を同時表示しない(枠間は最低 1 画面分のコンテンツ)。
- 広告がコンテンツ量を超えないページ(データが薄い選手ページ等)は枠 ① のみ。
- `beta/` 配下・`contact` / `privacy` / `about` は広告なし。

## 3. 自動広告の設定変更(AdSense 管理画面、コード変更不要)

- **ページ内フォーマット(自動挿入)をオフ**: 手動枠と重複し UX 崩壊の主因のため。
- **アンカー広告はオン**: モバイル収益への寄与が大きく、閉じるボタンがあり UX 影響が比較的小さい。
- **モバイル全画面(ビネット)は一旦オフ**: 収益は高いが離脱要因。手動枠の収益が安定した後に A/B で判断。
- **除外設定**: `/beta/*`, `/contact`, `/privacy`, `/about` を除外 URL に登録。
- **広告掲載率**: 100% から始めず 70〜80% に設定し、収益曲線を見ながら調整(掲載率を下げても収益は比例して落ちない)。

## 4. 計測・検証

- 広告ユニットをページ種別ごとに分け、AdSense レポートで RPM をユニット単位で比較。
- GA で広告導入前後の直帰率・滞在時間・ページ/セッションを比較(導入週と前週)。
- Search Console / CrUX で CLS を監視。導入後に CLS > 0.1 になったら min-height 設定を見直す。
- 1 枠ずつ段階導入し、UX 指標の劣化がないことを確認してから次の枠を追加する。

## 5. 実装ステップ(着手時)

1. AdSense 管理画面で自動広告設定を変更(§3)— コード変更なしで即日可能
2. AdSense 管理画面でディスプレイユニットを作成(players / teams / matches 用)
3. `AdUnit.tsx` 実装 + players/[id] に枠 ① のみ導入
4. 1〜2 週間計測 → 問題なければ teams / matches へ展開
5. `docs/wiki/monetization.md` の Open Questions(掲載面と広告ユニット設計)を更新
