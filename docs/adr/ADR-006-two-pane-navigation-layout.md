# ADR-006: 公開サイトのナビを左サイドバー型2ペイン（折りたたみ可能）へ刷新する

## Status

Accepted（高レベル方針）。決定日: 2026-06-22。実装は段階的に行い、詳細は
親仕様の残課題に残す。

親仕様: `docs/raw/2026-06-22-nav-two-pane-design.md`
現状仕様: `docs/wiki/public-pages.md`（「ナビゲーション再設計方針」節）

## Context

公開サイトのナビは `src/components/Header.tsx` の横並び1本のみで、モバイル向けの
折りたたみ（ハンバーガー）も未実装だった。一方、コンテンツは tournaments /
players / teams / highschool / st-league / news / growth と多階層に育ち、
グローバルナビが入口の一部しか露出していない。

ヒアリングで最優先要件は「大会 ↔ 選手 ↔ 年度 ↔ チームのように回遊が深まること」と
確認された。回遊を面で支えるには、入口を常時露出できる持続的なナビ面（サイドバー）と、
現在地に応じた局所ナビ、エンティティ間の相互リンクが要る。

後戻りコストが高いのは個々のリンク文言ではなく、全公開ページが乗るレイアウトの骨格
（シェル）と、既存の SEO 内部導線との関係である。ここを先に決める必要があった。

## Decision

公開サイト（`softeni-pick` mode）のレイアウトを、左サイドバー＋右コンテンツの
2ペインに刷新する。骨格として `AppShell`（上部バー＋サイドバー＋ドロワー）を新設し、
`_app.tsx` の `<Header/> + <main/>` 構成を置き換える。

確定した方針:

- PC: 左サイドバー＋右コンテンツの2ペイン。サイドバーは折りたたみ可能
  （ピン留め・状態保持）。
- モバイル/タブレット: 上部バーにハンバーガー、左ドロワーでサイドバーを開閉。
- `score` mode: サイドバーを出さず**上部バーのみ**。`isScoreSiteMode()` 分岐を踏襲。
- グローバル区分（サイドバー第1階層）は「セクション入口」に限定。
- コンテキスト第2階層は**本文上部のサブナビ**に置く。
- コンテンツ最大幅は現状最大（`max-w-6xl`）まで取れるシェルとし、サイドバーが
  コンテンツを狭めない。各ページの `maxWidth` 指定は維持。
- 年度前後ナビは全エンティティ共通の汎用コンポ（`YearPagerNav`・仮）に集約。
- 既存 SEO 内部導線を再利用・尊重し、重複リンク／パターン二重化／絞り込み意図の
  上書きを避ける。

## Alternatives

- A. ハンバーガー対応のみ（横並び維持）: 最小コストでモバイル課題は解くが、
  回遊を面で支える持続的ナビにならず、最優先要件を満たさない。
- C. ハイブリッド（上部バー＋ドロワーのみ）: B と A の中間。常設の入口露出が弱く、
  PC での回遊体験が B に劣る。
- 第2階層をサイドバー内に置く案: サイドバーが肥大化し、SEO 導線（本文側の
  セグメント型リンク）と二重化しやすいため、本文サブナビを採用。

## Consequences

メリット:

- 入口が常時露出し、セクション横断の回遊が容易になる。
- レイアウト骨格を `AppShell` に集約でき、ヘッダー/サイドバー/ドロワーの一貫管理。
- 年度前後ナビの汎用化で重複実装を削減。

デメリット・リスク:

- 全公開ページが乗るシェル変更のため影響範囲が広い（`_app.tsx` / `PageLayout` /
  `Header`）。段階導入で緩和する。
- 新ナビと既存 SEO 導線の重複により内部リンクが薄まるリスク → サイドバーは入口に
  限定し、末端リンクは本文導線に委ねる方針で回避。
- 折りたたみ状態保持・サブナビ共通化範囲・年度前後の存在判定データ源は実装時に詰める。
