# ADR-004: 成長分析の公開境界と同意レベル（段階公開モデル）

## Status
Draft（推奨デフォルトを提示。同意主体・未成年配慮の最終仕様は Open Questions に残す）。

## Context
`/beta/matches-results/growth`（および score mode `/matches/growth`）の成長分析は、
score 機能の延長として「部活などグループ内での成長記録」を想定して作られた。
この方針は残しつつ、softeni-pick 本体のコンテンツとして閲覧される機能へ育てたい。

現状の実装は次のようになっている（`lib/growthAnalysis.ts`）。

- `buildGrowthTargets(matches)` は公開試合に登場する**全選手/ペアを自動的に対象化**する。
- ターゲットキーは `player:<正規化氏名>` / `pair:<氏名&氏名>`（`getSideTargetBase`）。
- `generate-beta-matches-json.mjs` が全ターゲットの `targets.json` と
  `reports/{target}.json` を生成し、公開出力する。

つまり現状、成長分析には**同意・公開フラグが一切存在しない**。
「試合結果が公開されている」という事実だけを根拠に、本人の個別同意なしに
**実名の能力分析ページ**が自動生成・公開されている。

ここで効く論点は 2 つ。

1. ADR-003 が定義する match 単位の `visibility`（掲載試合=public）と、
   **個人の成長分析の公開可否は別物**である。試合結果の公開と、
   その人を名指しで「サーブが弱い/キーポイントに弱い」と継続分析・将来ランキング化することは、
   感度が質的に異なる。
2. ソフトテニスは中高生（未成年）の比率が高く、実名での能力評価・他者との横断比較・
   ランキング掲載はとくに慎重な扱いが要る。

本体コンテンツ化（プロフィール統合、横断ベンチマーク、ランキング/集約記事）へ進める前に、
公開境界と同意の方針を先に確定しないと、後続実装が手戻りする。

## Decision
成長分析の公開を一律にせず、**ターゲット（個人/ペア）単位の段階公開（visibility ラダー）**を導入する。
match 単位の `visibility`（ADR-003）とは**直交する独立軸**として扱う。
match が public でも、その個人の成長分析が自動で public になることはない。

### 1. 公開レベル（ラダー）
ターゲット単位に `growthVisibility` を持たせる。

- **L0 none** — レポートを公開出力しない。記録のみ。`targets.json` / `reports/*.json` に出さない。
- **L1 link** — `targetKey` を知る人だけ閲覧可。`targets.json` に**載せず**、レポートは
  直リンクのみ・`noindex`。部活内共有の既定面。
- **L2 public（自己分析のみ）** — 選手/ペアページに成長セクションを表示しインデックス対象。
  ただし**横断比較（他者・全体平均・パーセンタイル）は表示しない**。自分の過去との比較に限る。
- **L3 ranked** — 全体平均比較・パーセンタイル・「最も伸びた選手」等のランキング/集約コンテンツに
  **実名で掲載可**。

感度で 2 層に割り切る。**自分の過去との比較（〜L2）は低感度**（本人データであり試合結果は既に公開済み）。
**他者と並べる/晒す（L3）は質的に別物**として、常に明示 opt-in を必須にする。

### 2. 推奨デフォルト
- **既存ターゲット（過去データ）= L1（link）に倒す。** 遡及的な実名サイト公開を避ける安全側。
- **新規ターゲット = グループ（チーム）単位の公開ポリシーに従う。** 既定はグループ未設定時 L1。
- **L3 は対象に関わらず常に明示 opt-in。** デフォルトで L3 になることはない。
- **個人の撤回（オプトアウト）権は全レベルで優先。** 撤回があれば次ビルドで L0 相当に落とす。

### 3. 同意の主体（二層モデル）
記録入力者は実運用上グループ（部活/サークル）であり、全選手から個別同意を取るのは運用上重い。

- **グループ単位の公開ポリシー**（このグループの記録は Lx まで公開可）を一次同意とする。
- **個人の撤回権**を二次的に上書き可能とする（個人 > グループ）。
- **未成年配慮**: L3（実名で他者と並べる/ランキング）は保護者同意フラグ（`guardian_consent`）が
  立っている場合のみ許可する。フラグ無しは最大 L2 で頭打ち。
- **匿名化オプション**: 地域・カテゴリのみ表示／イニシャル表示を用意し、実名を出さずに
  集約コンテンツ（ランキング・伸び率）へ参加できる経路を残す。匿名参加は L3 でも保護者同意を不要にできる。

### 4. データモデル
Supabase に同意テーブルを新設する（match とは別軸なので match 行には持たせない）。

```
growth_consent
  subject_key      text   -- player:<正規化氏名> / pair:<...>（getSideTargetBase と同一規則）
  level            text   -- 'none' | 'link' | 'public' | 'ranked'
  consent_source   text   -- 'group' | 'self' | 'guardian'
  group_id         text   -- グループ起点同意の場合
  guardian_consent boolean default false
  anonymized       boolean default false
  granted_at       timestamptz
  withdrawn_at     timestamptz null
```

実効レベルの決定規則: `min(group_policy, self_override)` を基本とし、
撤回（`withdrawn_at` 有）は即 L0、`guardian_consent=false` は L2 で上限クランプ。

### 5. 生成パイプラインへの反映
- `buildGrowthReports` / `generate-beta-matches-json.mjs` で、実効レベル `< public` の
  ターゲットを `targets.json` から除外。`link` はレポート JSON のみ生成し `noindex`。
- L2 のレポートは横断比較ブロックを出力段で落とす（`toPublicMatchSnapshot` が内部項目を
  削るのと同じ「公開境界で削る」思想に揃える）。
- L3 限定で全体平均・パーセンタイル・ランキング集約を生成。匿名ターゲットは表示名を
  地域/カテゴリ/イニシャルに置換。
- 撤回は静的生成サイクルでのみ反映される（即時でない）。`data-import.md` の上限運用と同じ注意として明記。

### 6. score mode / softeni-pick mode の差
- softeni-pick mode: L2 以上をメディアとしてインデックス公開（ADR-003 の「閲覧公開＝本体統合」と整合）。
- score mode（UGC 本拠地、private 既定）: 既定は L1 相当。ユーザーが明示公開したものだけ共有 URL で見せる。

## Alternatives
- **現状維持（match 公開＝成長公開を継承）**: 実装は最小だが、実名能力評価を無同意で公開し続けることになり、
  未成年比率の高い競技で横断比較・ランキングへ拡張する前提と両立しない。却下。
- **全ターゲット opt-out（公開既定・撤回のみ）**: 導線は軽いが、既存データを遡及的に実名公開する初期状態が
  リスク。中間案として L3 のみ opt-in にする折衷も可能だが、初期既定は安全側（L1）を選ぶ。
- **match 行に成長公開フラグを同居**: 個人は複数 match にまたがるため、ターゲット単位の同意を
  match 行で表現すると不整合。別テーブル（subject_key 単位）が正しい。

## Consequences
- メリット: 「部活内の私的記録（L1）」と「本体の公開コンテンツ（L2/L3）」を同一エンジンで両立できる。
  ADR-003 の visibility 思想と一貫し、公開境界を生成段で一元的に絞れる。
- デメリット / 新規に必要なもの: `growth_consent` テーブル、実効レベル計算、生成スクリプトの絞り込み、
  L2 での横断比較の出し分け、匿名化レンダリング、グループ/個人/保護者の同意取得 UI と撤回導線。
- `GrowthTarget` 型に実効 `growthVisibility`（と `anonymized`）を追加する必要がある。
- 撤回反映が静的生成サイクル依存になるため、運用上の SLA をドキュメント化する。

## Open Questions
各項目は「やりたいこと → だから起きる問題 → だから決めること」で記述する。

### Q1. 同意主体はグループか本人か
- やりたいこと: 全選手から個別同意を取るのは運用上無理なので、入力者である部活（グループ）が
  「うちの記録は公開可」と一度設定すれば公開できるようにしたい（グループ一次同意）。
- 問題: 部活（顧問・代表）の「公開可」は、分析対象の選手本人が「自分の弱点を実名で世間に出される」ことへの
  同意とは限らない。顧問と本人の利害がずれる。本人同意なしに個人を名指し評価したデータを出し続けてよいかが曖昧。
- 決めること: どのレベルまでグループ同意だけで出し、どこから本人同意を必須にするか
  （例: L1 まではグループ同意で可、L2 以上は本人同意必須）。

### Q2. 未成年・保護者同意の取得と確認
- やりたいこと: コンテンツの中心は中高生なので、学生も L2/L3 で公開・ランキング掲載したい。
- 問題: 未成年は本人同意だけでは足りず保護者同意が要る場面がある。だが部活が一括入力する運用で、
  多数の保護者同意を集める現実的な手段がない。`guardian_consent` フラグを作っても、
  誰がどう立て、実際に保護者が同意したとどう確認するのかが不明。
- 決めること: 保護者同意を集め・確認する手段を作るのか。作れないなら
  「未成年は L3 を匿名のみに制限」と割り切って回避するのか。

### Q3. ランキング/集約での実名 vs 匿名の既定
- やりたいこと: 「最も伸びた選手」ランキングや伸び率の集約記事を作りたい（新規流入の目玉）。
- 問題: 実名のほうがコンテンツとして強く SEO も効くが、同意ハードルが上がり未成年問題に直撃する。
  匿名（地域・カテゴリ・イニシャル）なら同意は緩むが訴求は弱い。
- 決めること: 集約コンテンツの既定を実名にするか匿名にするか。推奨は「既定匿名＋実名は opt-in で昇格」だが、
  それで記事として成立する魅力が残るかを判断する。

### Q4. 既存ターゲットを L1 に倒した後の引き上げ導線
- やりたいこと: 既存データは安全側で L1 に倒すが、そのままでは公開コンテンツが空になるので、
  同意を取って L2 以上へ引き上げたい。
- 問題: 既に記録済みの過去の選手に「公開してよいか」を聞く連絡手段（メアド等）を持っていない。
- 決めること: 誰に・どの画面で・どうやって引き上げ同意を取るか
  （部活経由で声かけ／サイト上に「自分のページを公開申請」導線、など）。

### Q5. 撤回反映の SLA
- やりたいこと: 「やっぱり消して」に応える撤回（オプトアウト）を用意したい。
- 問題: サイトは静的生成のため、撤回を DB に書いても次のビルド＆デプロイまで実ページは消えない。
  炎上・いじめ・個人情報懸念で「今すぐ消したい」場合、ビルド周期では遅すぎる。
- 決めること: 撤回を何時間以内に反映すると約束するか。緊急削除用に該当 JSON 即時削除／CDN パージの
  別経路を持つか。

### Q6. score mode（UGC）転換時の同意モデル統合
- やりたいこと: ADR-003 Phase 2 で score 側を UGC（ログインユーザーが自分の試合を記録）の本拠地にする計画があり、
  そこでは選手＝ログイン本人。
- 問題: 本 ADR の `growth_consent` は `subject_key`＝正規化氏名のグループ起点。UGC では同意主体が
  `owner_user_id`（アカウント本人）になり、氏名ベースのグループ同意とアカウント同意の 2 モデルが併存・衝突する
  （同一人物が両方に存在した場合の優先順位など）。
- 決めること: 認証ユーザー所有モデル導入時に、`growth_consent` をどうアカウントへ紐づけ直すか（移行方針）。
