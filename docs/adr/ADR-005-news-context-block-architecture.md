# ADR-005: 速報・プレビュー機能を「文脈ブロック一次成果物 + イベント抽出」アーキテクチャで作る

## Status

Accepted（高レベル方針・データフロー）。具体ブロックの語彙・実装詳細は Open Questions と raw 設計に残す。

決定日: 2026-06-21。親仕様: `docs/raw/2026-06-21-news-auto-draft-design.md`、現状仕様: `docs/wiki/news-context-blocks.md`。

## Context

流入の大きい時事系（速報・プレビュー）は需要が大きいが、現行の手動入力中心の運用では即応が難しく、機能化が見送られてきた。一方で自動化すれば競合（汎用テンプレ SEO farm）も同じことができ、強みにならない懸念があった。

競合調査（2026-06-21）で、ソフトテニス速報領域は二極化していることが分かった。

- 汎用テンプレ SEO farm（zutto-sports.com / sposoku.com）: 全スポーツ横断のテンプレで大会ページを空作成→検索意図を先取り→結果を貼るだけ。速いが、選手・歴代・通算などの文脈はゼロ。構造化 DB を持たないため文脈を出せない。
- 編集・公式メディア（softtennis-mag.com / JSTA / 個人ブログ）: 深いが人力依存でスケール・速度が出ない。

「速くて、かつ DB 由来の文脈が付く」枠が空いており、そこは Softeni Pick の `data/tournaments/details/**` 中心の構造化データ（歴代・通算・節目）でしか埋められない。farm は原理的に持てず、編集メディアは人力で再現できない。

このとき設計の重心をどこに置くかが論点になった。素朴には「速報記事を生成する機能」と捉えがちだが、記事を頂点に置くと文脈が記事に従属し、大会ページ・選手ページ・ランキングなど他面で再利用しにくくなる。本機能で後戻りコストが高いのは、生成物の構造（何を一次成果物にするか）である。

## Decision

**文脈ブロックを一次成果物とし、その上位に「イベント抽出」レイヤを置く**。記事はブロックの再利用先の一つにすぎない、と位置づける。

採用するデータフロー:

```
大会データ（data/tournaments/details/**, players/index.json）
  ↓ イベント抽出（初優勝 / 連覇 / 王者撃破 / 通算N勝 / ベスト4初進出 など、意味のある出来事を構造化）
  ↓ 文脈ブロック生成（一次成果物。決定的・テンプレート）
  ↓ 再利用先：大会ページ / 選手ページ / ランキング / 記事（/news）
```

旧案「大会データ → 記事生成 → 文脈ブロック」は採らない。

### 付随して確定した方針

1. データ取得は完全手動入力のまま維持し、自動化するのは生成のみ。外部速報元の自動クロールはしない（保守・正確性・権利リスク回避）。
2. 本文生成は LLM を使わず**テンプレートのみ**。差別化は文章の自然さでなく DB 由来の事実密度で行う（誤り混入ゼロ・低コスト・鮮度シグナル安定）。
3. 公開記事（`/news/[articleId]`）は **human-in-the-loop**（自動ドラフト→人が承認→公開）。既存ページへのブロック差し込みは決定的生成のためビルド時自動（既存 `analysis.json` 生成と同格）。
4. ブロックは安全性・再利用価値で段階導入する。優先度A=`historical-winners`/`milestone`/`career-record`、B=`seed-vs-result`、C=`head-to-head`（名寄せ誤判定リスクが高く、精度検証まで導入しない）。
5. 実装順序は既存ページ強化を記事より先、B（プレビュー）を A（結果）より先にする。記事未完成でも既存ページの情報密度向上で SEO 効果を先取りするため。
6. 記事 URL は `/news/[articleId]` 独立ツリー。`/tournaments/.../preview` 等は既存大会ページとのカニバリ距離が近いため不採用。プレビュー記事は結果確定後に同一 URL（`articleId` 共有）で結果記事へ昇格させる。
7. `historical-winners` は既存 `lib/highschoolNationalTournaments.ts`（高校歴代の優勝〜ベスト4抽出）を大会非依存に一般化して実装する（新規ロジックを二重に作らない）。

## Alternatives

- **記事を一次成果物にする（旧案: データ→記事→ブロック）**: 実装は素直だが、文脈が記事に従属し他面で再利用できない。farm と同じ土俵（記事の速さ）で戦うことになり差別化資産が育たない。却下。
- **LLM で本文生成**: 文章は自然になるが、誤情報混入・レビューコスト・API コスト・ビルド日由来の鮮度ノイズが増える。本機能の価値は事実密度であり LLM の自然文は不要。却下。
- **外部速報元を自動クロールして即時性で勝負**: farm と同方式で速度は出るが、スクレイピング保守・正確性・権利のリスクを負い、かつ即時性は本機能の主価値ではない。却下。
- **イベント抽出を作らず具体ブロックだけ実装**: 短期は速いが、`milestone` 等の「出来事」を各面で個別実装することになり再利用が効かない。イベント列を一次データに持つ方が長期資産になる。段階導入（初期は具体ブロック、`milestone` を皮切りにイベント層へ寄せる）で折衷。

## Consequences

- メリット: 文脈ブロック／イベント列を大会・選手・ランキング・記事で再利用でき、farm が構造的に持てない差別化資産になる。記事機能の完成を待たず既存ページ強化で SEO 効果を先取りできる。テンプレート＋手動入力維持で運用負荷は「チェックのみ」に圧縮され、誤情報・保守リスクが低い。
- デメリット／新たに必要になるもの: イベント抽出・文脈ブロックの抽出ロジック（`lib/` 横断集計）が最重量。`milestone`／イベント語彙の定義、`articleId` 命名規約、承認フロー（当面は `state` 手書き、記事実装時に beta 管理面へ）が必要。
- SEO: 新ページ種別 `/news` を seo.md 重複マップ #8 に追加済み。結果記事 × 大会年度別ページのカニバリは公開後 GSC で実測して judge（farm とは「{大会}{年}結果速報」で正面勝負しない方針）。
- 既存資産との関係: `lib/highschoolNationalTournaments.ts` を汎用コア（`lib/tournamentRecords.ts` 仮称）＋高校固有ラッパに再編する。高校歴代ページの出力は不変（スナップショットで担保）。

## Open Questions

- `milestone` / イベント抽出の語彙確定（初優勝・連覇・3連覇・初出場・王者撃破・通算N勝・ベスト4初進出 のキー定義と判定条件）。
- イベント抽出レイヤを最初から薄く実装するか、`milestone` を内包する形で後付けするかの実装判断。
- `head-to-head` 導入可否を判断する名寄せ精度の検証方法（誤マッチ率の計測手順）。
- 記事 `articleId` の命名規約（プレビュー→結果で共有する安定 ID）。
- 大会改称をまたぐ歴代結合（エイリアス table を持つか tournamentId 単位で割り切るか）。
- 承認 UI の最終的な置き場（`state` 手書き → beta 管理面への移行時期）。
