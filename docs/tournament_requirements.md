# 大会一覧ページ 要件整理

## 1. 目的

-   過去のソフトテニス大会を一元的に把握できるサイトを構築する
-   結果データは今後も手動で拡張していく
-   「結果の完全保持」を前提にせず、大会の存在を整理する
-   大会を探せる"検索ハブ"を作る

------------------------------------------------------------------------

## 2. 前提条件

-   既存の大会ページ構造は変更しない
-   既存のURL体系は維持する
-   開催予定の特別セクションは不要（現時点では考慮しない）
-   一覧は日付順で並べる
-   フィルターで検索可能にする

------------------------------------------------------------------------

## 3. 全体構造

### 大会一覧ページ

/tournaments

→ 検索・一覧専用ページ（ハブ）

------------------------------------------------------------------------

### 主要大会一覧ページ

/tournaments/major

→ カテゴリ（世代）別に大会をカード表示する旧来ページ。既存URLとして維持。

------------------------------------------------------------------------

### 既存構造（維持）

/tournaments/\[generation\]/\[tournamentId\]/\[year\]\
/tournaments/\[generation\]/\[tournamentId\]/\[year\]/...

一覧ページからはこの既存ページへリンクする。

------------------------------------------------------------------------

## 4. 表示単位

一覧の単位は「大会マスター」ではなく「大会開催インスタンス」。

例：

-   2025年 北海道高校総体
-   2024年 北海道高校総体

同じ大会でも年が違えば別行として表示する。

------------------------------------------------------------------------

## 5. 並び順

-   デフォルト：開催日降順（新しい順）
-   ソートは日付基準のみでよい（当面）

------------------------------------------------------------------------

## 6. UI設計要件

### レイアウト構造

\[フィルターエリア\]\
\[件数表示・ソート\]\
\[大会一覧テーブル\]

------------------------------------------------------------------------

## 7. フィルター項目

-   都道府県（複数選択可）
-   種類（全国／ブロック／都道府県／市町村）
-   対象カテゴリ（中学／高校／一般／シニア）
-   年（単年）
-   結果データありのみ（内部データあり）

### フィルター仕様

-   即時反映型（Applyボタンなし）
-   URLクエリと連動

例：

/tournaments?prefecture=hokkaido&level=prefecture&year=2025

------------------------------------------------------------------------

## 8. 一覧表示形式

テーブル形式。

### 列構成（最小）

| 日程 \| 大会名 \| 地域 \| 種類 \| 対象 \| 結果 \|

------------------------------------------------------------------------

### 各列仕様

#### 日程

-   開始日を表示
-   終了日はサブ表示可（例：6/10--6/12）

#### 大会名

-   クリックで既存年度ページへ遷移
    /tournaments/\[generation\]/\[tournamentId\]/\[year\]

#### 地域

-   都道府県

#### 種類

-   バッジ表示

#### 対象

-   タグ表示（複数可）

#### 結果

-   内部保持あり → 表示
-   外部リンクのみ → アイコン表示
-   なし → 空

------------------------------------------------------------------------

## 9. UX設計方針

-   サイドバーは使わない（横フィルター）
-   テーブルヘッダーは sticky
-   行ホバーで強調
-   モバイルではカード表示に切替
-   セクション分割はしない

------------------------------------------------------------------------

## 10. データ構造（概念）

``` ts
interface TournamentInstance {
  generation: string
  tournamentId: string
  year: number
  startDate: string
  endDate: string
  prefecture: string
  level: "national" | "block" | "prefecture" | "city"
  categories: string[]
  hasInternalResult: boolean
}
```

リンク生成：

/tournaments/${generation}/${tournamentId}/\${year}

------------------------------------------------------------------------

## 11. 役割分離

### 検索ページ

大会を見つける場所。

### 既存ページ

大会の詳細・結果を見る場所。

責務を明確に分離する。

------------------------------------------------------------------------

## 12. 将来拡張可能性

この構造なら：

-   開催予定の追加
-   人気大会表示
-   カレンダー表示
-   他競技への拡張
-   API化

が可能。

------------------------------------------------------------------------

## 設計の本質

既存の結果アーカイブ構造は維持しつつ、
その上に「大会検索インターフェース」を追加する。

探索性を強化するレイヤー設計とする。
